# Audit: 2024-01-init-capital-invitational

## Missing Access Control on LsdApi3OracleReader Setters
- Location: `contracts/oracle/LsdApi3OracleReader.sol` : `setDataFeedIds`, `setQuoteTokens`, `setMaxStaleTimes`
- Mechanism: Unlike `Api3OracleReader` (which gates all setters with `onlyGovernor`) and unlike `LsdApi3OracleReader.setApi3OracleReader` (which has `onlyGovernor`), the functions `setDataFeedIds`, `setQuoteTokens`, and `setMaxStaleTimes` have **no access-control modifier**. Any arbitrary caller can overwrite the `dataFeedId`, `quoteToken`, or `maxStaleTime` for any token at any time.
- Impact: An attacker can point a token's `dataFeedId` to a malicious or stale API3 feed, set `quoteToken` to a token with a manipulated price, or set `maxStaleTime` to `type(uint96).max` to accept arbitrarily stale prices. Since `LsdApi3OracleReader.getPrice_e36` is used by `InitOracle` → `InitCore` for collateral and borrow credit calculations, an attacker can manipulate the reported price of LSD-backed collateral (e.g., wstETH), causing positions backed by that collateral to be overvalued. This enables stealing funds from the protocol by opening massively over-collateralized positions against worthless or manipulated collateral, or triggering unjustified liquidations of healthy positions.

## Missing Order Ownership Check in MarginTradingHook.updateOrder
- Location: `contracts/hook/MarginTradingHook.sol` : `updateOrder`
- Mechanism: `updateOrder` loads `initPosId = initPosIds[msg.sender][_posId]` and verifies it is non-zero (i.e., the caller owns *some* position), and uses it to fetch `marginPos` for the `collAmt` ceiling check. However, it **never verifies that `order.initPosId == initPosId`** — i.e., it never checks that the order being updated actually belongs to the caller's position. Compare with `cancelOrder`, which correctly includes `_require(order.initPosId == initPosId, Errors.INVALID_INPUT)`.
- Impact: Any user who has ever created a margin position can modify **any other user's active order**. An attacker can overwrite a victim's `triggerPrice_e36`, `limitPrice_e36`, `tokenOut`, and `collAmt`. For example, setting `limitPrice_e36 = 1` and `collAmt` to the maximum collateral would cause the victim's stop-loss/take-profit order, when filled by an arbitrager, to liquidate the victim's position at a catastrophically bad price, draining the victim's collateral to the arbitrager (or attacker acting as arbitrager).
