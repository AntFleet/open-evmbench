# Audit: 2026-01-tempo-feeamm

## Reentrancy in `burn` (state updates after external transfers)
- Location: `contracts/FeeAMM.sol` : `burn`
- Mechanism: The function transfers both `userToken` and `validatorToken` to `to` *before* decrementing `liquidityBalances[poolId][msg.sender]`, `totalSupply[poolId]`, and the pool reserves. The only guard against over-burning is `liquidityBalances[poolId][msg.sender] < liquidity`, which is checked before the external calls but is not effects-written until after them. If `to == msg.sender` (an attacker-controlled contract) and either token performs a recipient callback (ERC‑777, ERC‑1363, or any non-standard ERC20 with hooks), the attacker's callback can re-enter `burn` (or any other function) while `liquidityBalances` and reserves are still at their pre-call values, allowing the same liquidity to be redeemed multiple times until the contract's actual token balances are exhausted.
- Impact: An attacker who is an LP in a pool whose `userToken` or `validatorToken` supports transfer callbacks can drain the entire pool—both the user-token and validator-token side—far in excess of their pro-rata share, by re-entering `burn` repeatedly during the outgoing transfer.

## `mint` over-issues liquidity due to inconsistent discounting between mint and burn
- Location: `contracts/FeeAMM.sol` : `mint` (else-branch) / `_calculateBurnAmounts`
- Mechanism: When adding to an existing pool, `liquidity = (amountValidatorToken * _totalSupply) / (pool.reserveValidatorToken + (N * pool.reserveUserToken)/SCALE)`. The `userToken` side of the denominator is discounted by `N = 9985` (i.e., 0.9985×), so the denominator is smaller than the true pro-rata pool value (`V + U`). On burn, however, `_calculateBurnAmounts` uses the *undiscounted* reserves (`liquidity * reserveUserToken / totalSupply` and `liquidity * reserveValidatorToken / totalSupply`). Because minting prices `U` at 0.9985× while burning redeems `U` at 1×, a minter receives slightly more pool share than their `validatorToken` contribution warrants. An attacker can repeatedly mint a small amount of `validatorToken` and immediately burn the resulting liquidity, each cycle extracting roughly `0.0015 * U * ε / (V + 0.9985·U)` of value from existing LPs.
- Impact: A cyclic mint→burn attack lets an attacker extract value from existing LPs (up to ~0.15% of the pool's `userToken` reserve), particularly when the pool is imbalanced with large `U` relative to `V` (the normal state after fee swaps have run). This is a persistent accounting bleed against passive LPs.

## Permissionless pool creation with arbitrary / untrusted tokens
- Location: `contracts/FeeAMM.sol` : `mint` (and `executeFeeSwap`, `rebalanceSwap`)
- Mechanism: There is no whitelist or factory gate on `userToken`/`validatorToken`. Anyone may call `mint` to instantiate a pool keyed by `keccak256(userToken, validatorToken)` using tokens they control. Because `executeFeeSwap` and `rebalanceSwap` perform fixed-price (≈1:1) swaps against whatever reserves exist, a pool that pairs a real stablecoin with an attacker-minted worthless token allows the attacker to mint themselves unlimited "validatorToken", provide initial liquidity, and then swap worthless tokens for real user tokens at the 0.997/0.9985 fixed ratios. Any user (or integrating contract) that targets the wrong pool ID will trade real value for worthless tokens.
- Impact: Phishing/fake-pool attacks: users or downstream integrations that interact with attacker-created pools lose real stablecoins for worthless tokens. The contract provides no on-chain mechanism to distinguish legitimate pools from spoofed ones.

## Unchecked ERC20 transfer/transferFrom return values
- Location: `contracts/FeeAMM.sol` : `executeFeeSwap`, `rebalanceSwap`, `mint`, `burn`
- Mechanism: All token interactions use `IERC20(...).transfer` / `transferFrom` without verifying the boolean return. Some legitimate stable tokens (e.g., USDT) return `void` rather than `bool`, and various non-standard tokens can return `false` on failure without reverting. In this contract a silent `false` from `transferFrom` in `mint`/`executeFeeSwap`/`rebalanceSwap` would still credit reserves/liquidity, and a silent `false` from `transfer` in `burn`/`executeFeeSwap` would still debit reserves—creating phantom reserves or letting users "swap" without tokens actually moving.
- Impact: With non-standard tokens, reserves and liquidity accounting can diverge from actual balances, enabling inflated liquidity minting, free withdrawals, or permanent accounting corruption of a pool.

## Solidity 0.7.x arithmetic with no overflow protection
- Location: `contracts/FeeAMM.sol` : `mint` (else-branch multiplication), `pool.reserveValidatorToken += ...` in `mint`/`executeFeeSwap`
- Mechanism: The contract is compiled under `pragma >=0.7.6 <0.8.0`, which has no built-in overflow checking. In `mint`'s else-branch, `amountValidatorToken * _totalSupply` is computed *before* `_requireU128(amountValidatorToken)` is enforced; if `_totalSupply` ever exceeds 2¹²⁸ (achievable by accumulating many large mints, each allowed up to 2¹²⁸−1), the product silently wraps. Separately, `pool.reserveValidatorToken += uint128(amountValidatorToken)` (and the analogous update in `executeFeeSwap`) can wrap a `uint128` reserve without any check, unlike `rebalanceSwap` which explicitly guards its `uint128` addition.
- Impact: Under pathological reserve/totalSupply conditions, overflow can produce incorrect liquidity or wrapped reserves, breaking pool accounting and enabling redemption of more tokens than entitled. Realistically only reachable with extremely large accumulations, but the absence of SafeMath/0.8 checks makes the code fragile.
