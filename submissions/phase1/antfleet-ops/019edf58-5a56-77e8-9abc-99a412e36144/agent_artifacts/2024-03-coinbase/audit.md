# Audit: 2024-03-coinbase

## MagicSpend: Missing asset validation in `withdraw()`
- Location: `src/MagicSpend/MagicSpend.sol` : `withdraw()`
- Mechanism: `validatePaymasterUserOp()` enforces `withdrawRequest.asset == address(0)` via the `UnsupportedPaymasterAsset` check, but the standalone `withdraw()` function omits this check entirely. It calls `_withdraw(withdrawRequest.asset, msg.sender, withdrawRequest.amount)`, which will transfer arbitrary ERC-20 tokens held by the contract if the owner signs a request with a non-zero `asset` field. Since the contract is documented as "Only ETH (associated with zero address) is supported for now," any ERC-20 tokens that end up in the contract (accidentally or otherwise) can be drained through `withdraw()` with a signed request whose `asset` is set to that token address.
- Impact: An attacker who obtains (or whose signing backend produces) a withdraw request with a non-zero `asset` can drain arbitrary ERC-20 token balances from the paymaster contract via the direct `withdraw()` path, bypassing the ETH-only restriction enforced on the paymaster flow.

## MagicSpend: Nonce burned and event emitted before signature/expiry validation
- Location: `src/MagicSpend/MagicSpend.sol` : `withdraw()` and `validatePaymasterUserOp()` (both call `_validateRequest` before signature/expiry checks)
- Mechanism: `_validateRequest()` irreversibly marks the nonce as used and emits `MagicSpendWithdrawal` before the signature or expiry is verified. In `withdraw()`, the order is: `_validateRequest` → signature check → expiry check. If the signature is invalid or the request is expired, the nonce is still consumed (though the subsequent revert in `withdraw()` rolls back state, so this only affects `validatePaymasterUserOp`). In `validatePaymasterUserOp`, the function does NOT revert on signature failure—it returns `validationData = 1`. This means the nonce marking and `_withdrawableETH` credit persist even when the signature is invalid, and the event is emitted for a withdrawal that will not occur. While the EntryPoint's account-validation gate limits external griefing, any path that reaches `validatePaymasterUserOp` with a valid account signature but invalid paymaster signature will permanently burn the user's nonce.
- Impact: A user's withdraw request nonce can be permanently consumed without a successful withdrawal, rendering the signed request unusable. Additionally, misleading `MagicSpendWithdrawal` events are emitted for withdrawals that never happen, breaking off-chain indexing and reconciliation.

## MagicSpend: `postOp` silently drops ETH when transfer fails
- Location: `src/MagicSpend/MagicSpend.sol` : `postOp()`
- Mechanism: `postOp` deletes `_withdrawableETH[account]` and then uses `SafeTransferLib.forceSafeTransferETH` with a fixed gas stipend (`GAS_STIPEND_NO_STORAGE_WRITES`, typically 2300 gas). `forceSafeTransferETH` wraps the `call` in `pop()`, silently ignoring failures. If the recipient account's `receive`/`fallback` function requires more than 2300 gas (or reverts), the ETH transfer silently fails. The storage has already been mutated—`_withdrawableETH[account]` is deleted—so the user's funds are irrecoverably lost from their accounting perspective, even though the ETH remains in the contract.
- Impact: If a user's account cannot accept ETH within the 2300-gas stipend (e.g., a smart-contract account with non-trivial receive logic), the user permanently loses their withdrawable ETH balance without receiving the funds.

## MagicSpend: `assert` in `postOp` prevents recovery from first-postOp revert
- Location: `src/MagicSpend/MagicSpend.sol` : `postOp()`
- Mechanism: `postOp` begins with `assert(mode != PostOpMode.postOpReverted)`. If the first `postOp` call reverts for any reason (e.g., arithmetic underflow on `maxGasCost - actualGasCost` if actual gas exceeds the estimate, or an out-of-gas in the ETH transfer), the EntryPoint calls `postOp` a second time with `PostOpMode.postOpReverted`. The `assert` then triggers a hard revert, preventing the EntryPoint's standard post-op-reverted error-handling flow from executing properly. This can cause the entire UserOperation bundle to fail rather than gracefully degrading.
- Impact: Any revert in the first `postOp` invocation cascades into an unrecoverable assert-revert on the second call, potentially failing the entire EntryPoint bundle and locking the user out of their withdrawable funds until they resort to the direct `withdrawGasExcess()` path.

## CoinbaseSmartWallet: Cross-chain replayable `upgradeToAndCall` via `executeWithoutChainIdValidation`
- Location: `src/SmartWallet/CoinbaseSmartWallet.sol` : `validateUserOp()` / `executeWithoutChainIdValidation()` / `canSkipChainIdValidation()`
- Mechanism: `canSkipChainIdValidation` whitelists `upgradeToAndCall` as a selector that may be executed without chain-ID binding. When a UserOperation's calldata selector matches `executeWithoutChainIdValidation`, `validateUserOp` recomputes the hash without the chain ID and requires `REPLAYABLE_NONCE_KEY`. This means a single valid signature for an `upgradeToAndCall` operation can be replayed across every chain where the same wallet address exists. If the upgrade is malicious (e.g., a phishing-signed upgrade to a malicious implementation on one chain), it automatically applies to all chains simultaneously with no per-chain opt-out.
- Impact: An attacker who obtains a single cross-chain-replayable signature for `upgradeToAndCall` can upgrade the wallet to a malicious implementation on every chain at once, fully compromising all cross-chain instances of that wallet.

## FCL: `ecAff_isOnCurve` does not reject the point (0, √b)
- Location: `src/FreshCryptoLib/FCL.sol` : `ecAff_isOnCurve()`
- Mechanism: The function rejects `(x=0, y=0)` and `x == p` / `y == p`, but does not reject `(x=0, y=√b mod p)`. For secp256r1, if `b` is a quadratic residue modulo `p`, then `(0, √b)` is a valid curve point that passes the on-curve check. While this particular point has order `n` (since the curve has prime order and cofactor 1) and thus does not create a subgroup-attack vector, accepting `x=0` as a public key creates an edge case in `ecZZ_mulmuladd_S_asm` where the doubling formula computes `M = 3*(X-ZZ)*(X+ZZ) = 3*(-ZZ)*(ZZ)`, which is valid but could interact unexpectedly with the point-at-infinity detection logic (`zz == 0` checks) if intermediate doublings produce zero-coordinate edge cases.
- Impact: Low direct impact for secp256r1 ECDSA verification because the attacker does not control the public key `Q`. However, if this library is reused in a context where the public key is attacker-supplied and additional validation is expected (e.g., key agreement), the acceptance of unusual points could lead to protocol-level vulnerabilities.
