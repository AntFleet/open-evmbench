# Audit: 2024-05-arbitrum-foundation

I found four genuine security issues.

## One-step proof `LOCAL_SET` does not update machine state
- Location: `src/osp/OneStepProver0.sol` : `executeLocalSet`
- Mechanism: `executeLocalSet` reads the top stack frame with `StackFrame memory frame = mach.frameStack.peek()` and updates `frame.localsMerkleRoot`, but `frame` is only a memory copy. The modified locals root is never written back into `mach.frameStack`, so the on-chain one-step transition for `LOCAL_SET` differs from the real WASM machine semantics.
- Impact: A challenger can prove and confirm a step using an invalid machine transition. Since `confirmEdgeByOneStepProof` relies on this prover to decide disputed assertions, this can let an attacker confirm an incorrect challenge edge and invalidate the honest assertion path.

## Force inclusion ignores timestamp delay
- Location: `src/bridge/SequencerInbox.sol` : `forceInclusion`
- Mechanism: `forceInclusion` reconstructs the delayed message preimage and enforces only `l1BlockAndTime[0] + delayBlocks_ < block.number`. It imports `ForceIncludeTimeTooSoon` but never checks `l1BlockAndTime[1] + delaySeconds` against `block.timestamp`, even though the inbox time variation model includes both block and timestamp bounds.
- Impact: Anyone can force-include delayed messages once the block delay passes even if the configured timestamp delay has not elapsed. This shortens the intended sequencer-only window and can force delayed messages into the L2 inbox earlier than the protocol’s time bounds allow.

## Allowlist can be bypassed through `tx.origin`
- Location: `src/bridge/AbsInbox.sol` : `onlyAllowed`
- Mechanism: The allowlist checks `isAllowed[tx.origin]` instead of authorizing `msg.sender`. If an allowlisted EOA is tricked into calling an unallowlisted contract, that contract can call inbox entrypoints and pass the allowlist check because the origin is allowlisted.
- Impact: When `allowListEnabled` is intended to restrict who can submit L2 messages, an attacker can route calls through a malicious contract and submit arbitrary inbox messages under that contract’s L2 alias using a phished allowlisted origin.

## Unattributed prefunded ERC20 inbox balance can be stolen
- Location: `src/bridge/ERC20Inbox.sol` : `_deliverToBridge`
- Mechanism: `_deliverToBridge` treats the inbox contract’s existing native-token balance as globally available. If `IERC20(nativeToken).balanceOf(address(this)) >= tokenAmount`, the caller is not charged at all; the bridge then pulls `tokenAmount` from the inbox and credits the message generated by the caller.
- Impact: Any native tokens sitting in `ERC20Inbox` before a call can be consumed by the next caller. An attacker can call `depositERC20` or create a retryable ticket using someone else’s prefunded or accidentally transferred inbox balance and receive the corresponding L2 credit.
