# Audit: 2024-01-renft

## Active hooks bypass core Safe Guard transaction checks, allowing theft of rented assets
- Location: `src/policies/Guard.sol` : `checkTransaction`
- Mechanism: When validating a Safe transaction, the contract checks if an active hook is set for the destination `to` address. If one exists, it delegates validation to the hook via `_forwardToHook(...)` and critically **skips** executing the default `_checkTransaction(...)`. The `_checkTransaction(...)` function enforces the protocol's core security invariants: it blocks malicious selectors like `transferFrom`, `approve`, and module/guard tampering functions. Because execution ignores this function, an asset or Safe loses all default protections if a hook targets it unless the hook meticulously re-implements them. For example, if a token contract is assigned a hook to permit claiming an airdrop or taking a game action, a renter can simply call `transferFrom` or `approve` to steal the NFT in active rental. The restricted call succeeds because the hook will ignore the unhandled transfer selectors instead of reverting, and `_checkTransaction` was completely bypassed. 
- Impact: A malicious renter can steal rented assets or permanently compromise the Safe's configuration, leading to a complete loss of lenders' funds.

## `DelegateCall` to batching contracts (e.g., MultiSend) trivially bypasses token transfer restrictions
- Location: `src/policies/Guard.sol` : `checkTransaction` / `_checkTransaction`
- Mechanism: The fundamental checks in `_checkTransaction` only inspect the outermost function selector (`mload(add(data, 0x20))`) of the transaction executed by the Safe. Gnosis Safe standardly relies on the `MultiSend` contract loaded via `DelegateCall` to batch multiple operations natively into one user transaction. If `MultiSend` (or any similar batching utility) is authorized into the protocol's whitelist (allowed by the `STORE.whitelistedDelegates(to)` check to provide essential grouped UX steps), a malicious renter can bypass token transfer protections by embedding blocked calls (like `transferFrom` or `approve`) inside the multicall payload. The outermost selector will evaluate as `multiSend(bytes)`, which comfortably passes the guard since it operates on an approval-basis for blacklisted selectors. The `MultiSend` contract will then faithfully unpack and execute the inner malicious operations unhindered.
- Impact: Renters can bypass all single-call token transfer blacklist checks and trivially steal in-rental NFTs by nesting transfers inside whitelisted batched transactions.
