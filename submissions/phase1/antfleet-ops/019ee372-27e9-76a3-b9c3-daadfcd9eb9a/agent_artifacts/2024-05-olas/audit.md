# Audit: 2024-05-olas

## Incorrect accounting in `buOLA.sol` `revoke()` leads to theft of other users' funds
- Location: `buOLA.sol` : `revoke`
- Mechanism: In `buOLA.sol`, the `revoke()` function is designed to cancel an ongoing lock, allowing the user to withdraw whatever has currently vested while burning the unvested remainder. To do this, it evaluates `_releasableAmount(lockedBalance)` to find the currently newly vested (pending) amount. However, it explicitly overwrites `lockedBalance.amountReleased = uint96(amountRelease);` with this delta rather than adding it to the cumulative `amountReleased`. Since `_releasableAmount()` only returns the previously unclaimed vested delta (total vested minus previously released tokens), overwriting the cumulative tracker destroys the record of prior withdrawals. When the user later calls `withdraw()`, the system calculates `amountBurn = lockedBalance.amountLocked - lockedBalance.amountReleased` and burns that inflated amount of OLA, while additionally transferring out the delta. This causes the contract to consume more OLA than what genuinely remained for the user.
- Impact: It causes the protocol to double-spend funds by erroneously burning and transferring more underlying OLA than an individual user has remaining. Since the contract holds pooled deposits from multiple users, it will consume OLA balances belonging to completely unrelated users. This essentially causes a loss and permanent destruction of tokens locked by innocent parties.
