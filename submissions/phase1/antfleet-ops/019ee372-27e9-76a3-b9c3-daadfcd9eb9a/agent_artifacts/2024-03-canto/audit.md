# Audit: 2024-03-canto

Here are the vulnerabilities found in the codebase:

## Missing caller verification in `lzCompose` allows attackers to steal user funds
- **Location:** `ASDRouter.sol` : `lzCompose()`
- **Mechanism:** The `lzCompose` function acts as the receiver for LayerZero's OFT composed messages. However, it lacks crucial access control checks to verify that `msg.sender` is actually the trusted LayerZero Endpoint. Because of this, anyone can call `lzCompose()` directly and pass entirely spoofed parameters. By calling it with an invalid payload (e.g., `composeMsg.length != 224`), an attacker triggers the `_refundToken` block. By placing the victim token address in `_from`, and specifying an arbitrary `amountLD` alongside their own address as `composeFrom`, the function blindly transfers the requested tokens out of the router contract to the attacker.
- **Impact:** An attacker can monitor the mempool for OFT bridging transactions and front-run the legitimate `lzCompose` callback, draining the bridged tokens from the `ASDRouter`. They can also steal any other stranded ERC20 tokens residing in the contract.

## `withdrawCarry` perpetually reverts due to invalid `view` declaration forcing a `STATICCALL`
- **Location:** `ASDOFT.sol` : `withdrawCarry()` and `CTokenInterfaces.sol` : `CTokenInterface`
- **Mechanism:** In `withdrawCarry()`, the contract attempts to calculate the protocol's accrued revenue by calling `CTokenInterface(cNote).balanceOfUnderlying(address(this))`. In `CTokenInterfaces.sol`, this function is incorrectly defined with the `view` modifier, forcing Solidity to execute it using `STATICCALL`. In compound-like protocol operations, `balanceOfUnderlying` is a state-mutating function (it calls `accrueInterest()` underneath to correctly timestamp the state before returning balances). Any attempt to execute a state-mutating instruction over a `STATICCALL` results in a hard EVM exception.
- **Impact:** The `withdrawCarry` function will always crash and revert on the very first instruction. The contract owner will be completely locked out of withdrawing the accrued yield generated by the NOTE reserves. 

## Unsafe downcasting on `minAmountASD` checking causes integer underflow / incorrect slippage validation
- **Location:** `ASDRouter.sol` : `_swapOFTForNote`
- **Mechanism:** When validating the slippage check, the function casts `_minAmountNote` to its signed version via `int minAmountInt = int(_minAmountNote)`. Immediately after, it takes the negated flow (`-baseFlow` or `-quoteFlow`) and compares it against `minAmountInt`. If `_minAmountNote` is sufficiently large (which realistically requires `uint` max-bounds values, but it's a completely user-supplied field under `payload._minAmountASD`), casting it to `int` will overflow into a severe negative number. 
- **Impact:** A malicious user or an endpoint manipulation attempt could supply a specifically crafted `minAmountASD` that bypasses the slippage check altogether due to the negative-value inversion, exposing the protocol or users to MEV sandwich attacks rendering the intended limit protection completely useless.
