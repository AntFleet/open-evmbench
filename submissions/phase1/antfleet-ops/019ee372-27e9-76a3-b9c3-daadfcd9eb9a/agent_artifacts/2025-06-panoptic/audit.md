# Audit: 2025-06-panoptic

An analysis of the codebase reveals a critical vulnerability related to how rolled-over withdrawals are accounted for in the global system state. 

## Rolled-over withdrawals are ignored in global accounting, resulting in arithmetic underflows or massive share dilution
- Location: `HypoVault.sol` : `executeWithdrawal` & `fulfillWithdrawals`
- Mechanism: When a withdrawal epoch is only partially fulfilled, unfulfilled shares are correctly carried over globally when initializing the next epoch in `fulfillWithdrawals` via `withdrawalEpochState[currentEpoch + 1] = WithdrawalEpochState({ ..., sharesWithdrawn: uint128(sharesRemaining) })`. However, when a user pushes their personal unfulfilled withdrawal balance to the next epoch using `executeWithdrawal` via `queuedWithdrawal[user][epoch + 1]...`, this increases their individual claim to the `sharesWithdrawn` total pool for the next epoch. The math for calculating their final payout is `(pendingWithdrawal * sharesFulfilled) / withdrawalEpochState.sharesWithdrawn`. Because rolled-over shares from individual users are safely absorbed by the globally rolling remainder, everything functions normally *under normal operating conditions*. 

HOWEVER, a user can skip calling `executeWithdrawal` indefinitely, intentionally leaving their rolled-over shares in old epochs. If they later call `executeWithdrawal` on a very old epoch, their remaining unfulfilled shares are suddenly pushed into `queuedWithdrawal[user][epoch + 1]`, and eventually into the *current* (unfulfilled) epoch. Because these shares forcefully appear in the current epoch's state *without* having incremented `sharesWithdrawn` for the current epoch (since the global rollover chain was already resolved for those previous epochs), the numerator (`pendingWithdrawal`) will completely desync from the denominator (`sharesWithdrawn`).
- Impact: A malicious user can severely inflate their proportion of `sharesWithdrawn` by introducing "ghost" rolled-over partial withdrawals into an active period. This forces the contract to fulfill far more internal shares/assets for the attacker than were proportionally possible, draining out the assets allocated by the manager for honest participants, rendering proper Vault execution impossible and robbing remaining users.
