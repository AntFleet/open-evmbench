# Audit: 2024-03-abracadabra-money

An analysis of the codebase reveals the following critical security vulnerability:

## Improper Lock Index Tracking During Expirations
- Location: `src/staking/LockingMultiRewards.sol` : `processExpiredLocks`
- Mechanism: The function allows an operator to process expired lock arrays, popping them and using swapping to efficiently reorganize the active set of locks. It correctly tracks the swapped index using `if (lastLockIndex[user] == lastIndex) lastLockIndex[user] = index;`. However, a vulnerability exists if an element pointed to correctly by `lastLockIndex` is inherently swapped due to *another* item being deleted such that `lastIndex` shifts downward without updating `lastLockIndex` if `lastLockIndex` became numerically equal to the new `lastIndex` *after* the previous pop. Wait, actually, if a user has locks `[A, B, C, D]` and `lastLockIndex` points to `D` (index 3), and `index=1` is deleted, `D` is swapped to index 1, and `lastLockIndex` goes to 1. The array length drops to 3 `[A, D, C]`. If `index=0` (`A`) is then deleted, `lastIndex` evaluates to 2 `C` and `C` swaps to 0, array shrinks to `[C, D]`. The problem is that index 2 (`C`) swaps over. But note that `lastLockIndex` remains `1` pointing to `D`. 

The real vulnerability is in how `lastLockIndex` tracking is handled when `index == lastIndex`, meaning the item being deleted is *already at the end of the array* (but is NOT `lastLockIndex`), which shrinks the array but does not change where `lastLockIndex` points, causing an out-of-bounds array access if `lastLockIndex` currently points to the index that will become entirely pruned in the future due to an array truncation that overlaps with `lastLockIndex`. Because the array shrinks, `locks[lastLockIndex]` may eventually be an out-of-bounds pointer or point to corrupted structured elements after resizing down, which leads to DoS anytime the user calls `stake` or `lock`.
- Impact: A user could have their subsequent `stake()` or `lock()` transactions permanently DoS'ed due to an array out-of-bounds error on `_userLocks[user][_lastLockIndex].unlockTime`, stranding their ability to accrue future boosted incentives and effectively bricking their staking integration.
