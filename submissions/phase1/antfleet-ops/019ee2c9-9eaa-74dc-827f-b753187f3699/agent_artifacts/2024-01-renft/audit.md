# Audit: 2024-01-renft

## Rental order hash does not bind `rentalWallet`
- Location: [smart-contracts/src/packages/Signer.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/packages/Signer.sol:162) : `_deriveRentalOrderHash`; [smart-contracts/src/policies/Stop.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/policies/Stop.sol:265) : `stopRent` / `stopRentBatch`; [smart-contracts/src/packages/Reclaimer.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/packages/Reclaimer.sol:71) : `reclaimRentalOrder`
- Mechanism: `RentalOrder`’s type string includes `rentalWallet`, but `_deriveRentalOrderHash` omits it from the actual `abi.encode(...)`. The stored `orders[orderHash]` flag is therefore not tied to the wallet that really holds the rented assets. `Stop` then trusts the caller-supplied `order.rentalWallet` for both `_reclaimRentedItems` and the rental-asset bookkeeping update. Because `reclaimRentalOrder` only checks that the delegatecall is executing in `rentalOrder.rentalWallet`, an attacker can reuse a valid stored order hash while swapping in a different protocol safe.
- Impact: A lender controlling any stoppable order can redirect reclaim execution into another rental safe that holds the same ERC1155 asset and pull that victim safe’s tokens to themselves; the same forged stop also decrements the victim safe’s rental blocklist, corrupting bookkeeping and potentially leaving the real rental unprotected or unrecoverable.

## Safe module removal checks the wrong calldata parameter
- Location: [smart-contracts/src/libraries/RentalConstants.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/libraries/RentalConstants.sol:62) : `gnosis_safe_disable_module_offset`; [smart-contracts/src/policies/Guard.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/policies/Guard.sol:255) : `_checkTransaction`
- Mechanism: The guard uses offset `0x24` for `disableModule(address prevModule, address module)`, which loads `prevModule`, not `module`. The whitelist check therefore validates the predecessor in Safe’s module linked list instead of the module being disabled. A renter can first enable any whitelisted extension, then disable a critical non-whitelisted module while presenting the whitelisted extension as `prevModule`.
- Impact: A renter can remove the `Stop` policy module from their safe, after which protocol stops cannot execute `execTransactionFromModule` to reclaim assets. That permanently locks rented assets in the renter’s safe.

## Rentals can be created without the corresponding transfers ever happening
- Location: [smart-contracts/src/policies/Create.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/policies/Create.sol:530) : `_rentFromZone`; [smart-contracts/src/policies/Create.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/policies/Create.sol:691) : `_executionInvariantChecks`; [smart-contracts/src/modules/PaymentEscrow.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/modules/PaymentEscrow.sol:361) : `increaseDeposit`
- Mechanism: `_executionInvariantChecks` only iterates over `zoneParams.totalExecutions` and checks recipients for whatever Seaport reports there; it never proves that every promised rental asset and payment actually produced an execution. `_rentFromZone` then derives `items` from the declarative `offer` and `consideration` arrays and unconditionally calls `STORE.addRentals(...)` and `ESCRW.increaseDeposit(...)`. If fulfillment can be arranged so expected transfers are omitted from `totalExecutions` (for example via self-directed or netted match paths), the protocol still books a live rental and a fake escrow deposit.
- Impact: An attacker can create phantom rentals backed by no real asset/payment movement, then stop them later and have `PaymentEscrow` pay out real ERC20 balances already sitting in escrow, draining funds from honest rentals.

## Extra consideration “tips” are treated as authenticated escrow payments
- Location: [smart-contracts/src/policies/Create.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/policies/Create.sol:326) : `_processBaseOrderConsideration`; [smart-contracts/src/policies/Create.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/policies/Create.sol:530) : `_rentFromZone`; [smart-contracts/src/modules/PaymentEscrow.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/modules/PaymentEscrow.sol:100) : `_safeTransfer`
- Mechanism: For BASE orders, every entry in `zoneParams.consideration` is blindly converted into a protocol payment item; the code does not distinguish original consideration from Seaport tip extensions, and the protocol signature does not bind those extra tip items. `_rentFromZone` then increases the escrow deposit for every such ERC20 item. A fulfiller can append an arbitrary ERC20 tip that transfers into escrow during creation but reverts or returns false on later `transfer`.
- Impact: Any fulfiller can hijack an otherwise valid BASE rental so that `stopRent` always reverts during payment settlement, permanently trapping the lender’s NFTs in the renter safe.

## Partial fills can collide onto one storage slot and strand the rest
- Location: [smart-contracts/src/packages/Signer.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/packages/Signer.sol:162) : `_deriveRentalOrderHash`; [smart-contracts/src/policies/Create.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/policies/Create.sol:578) : `_rentFromZone`; [smart-contracts/src/modules/Storage.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/modules/Storage.sol:189) : `addRentals` / `removeRentals`
- Mechanism: Active rentals are keyed by a single boolean `orders[orderHash]`. The derived hash contains no fill nonce or fill-instance identifier, so identical partial fills of the same Seaport order can produce the same rental hash. `addRentals` just sets the same boolean to `true` again, while `removeRentals` deletes it on the first stop. Any remaining slice still held by the safe/escrow no longer has a corresponding active order entry.
- Impact: A renter can split a partially-fillable order into multiple identical fills and cause the first stop to invalidate the shared order slot, after which later slices cannot be stopped. The remaining rented NFTs and escrowed payments become stuck.

## Overlapping reward-hook rentals can redirect previously accrued rewards
- Location: [smart-contracts/src/examples/revenue-share/ERC20RewardHook.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/examples/revenue-share/ERC20RewardHook.sol:75) : `onStart` / `onStop`; [smart-contracts/src/examples/revenue-share/ERC1155RewardHook.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/examples/revenue-share/ERC1155RewardHook.sol:81) : `onStart` / `onStop`
- Mechanism: Both hooks aggregate state only by `keccak256(safe, token, identifier)`. They do not persist the lender or the split parameters for existing rentals, but when a new overlapping rental starts or stops they accrue rewards for the entire `currentAmount` using the latest caller-supplied `RevenueShare`. That lets a later rental retroactively choose who receives rewards that were earned while earlier lenders’ inventory was active.
- Impact: By opening or closing a small overlapping rental on the same asset key, an attacker can siphon accrued rewards away from prior lenders and into an attacker-controlled address.

## `OrderMetadata` signatures do not bind `orderType` or `emittedExtraData`
- Location: [smart-contracts/src/packages/Signer.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/packages/Signer.sol:218) : `_deriveOrderMetadataHash`; [smart-contracts/src/policies/Create.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/policies/Create.sol:626) : `_isValidOrderMetadata`; [smart-contracts/src/policies/Create.sol](/Users/augstar/open-evmbench/audit_sources/2024-01-renft/smart-contracts/src/policies/Create.sol:165) : `_emitRentalOrderStarted`
- Mechanism: The declared `OrderMetadata` type includes `orderType` and `emittedExtraData`, but `_deriveOrderMetadataHash` hashes only `rentDuration` and the hook hashes. That truncated hash is used both as the Seaport `zoneHash` check and inside the protocol-signed `RentPayload`. So two security-sensitive metadata fields are mutable by the fulfiller even though the protocol treats the payload as signed.
- Impact: A fulfiller can inject arbitrary `emittedExtraData` into `RentalOrderStarted` and present metadata that neither the lender nor the protocol signer actually approved. Current item-shape validation limits direct on-chain `orderType` abuse, but the authentication boundary is already broken.
