# Audit: 2025-02-thorwallet

Here is the security audit of the provided Solidity codebase.

## 1. Permanent Denial of Service (DoS) in `withdrawRemainingTitn` due to Arithmetic Underflow
- **Location:** `MergeTgt.sol` : `withdrawRemainingTitn()`
- **Mechanism:** In `withdrawRemainingTitn()`, the contract captures the initial parameters at the first call after 360 days: `remainingTitnAfter1Year` is set to the current TITN balance, and `initialTotalClaimable` is set to `totalTitnClaimable`. It then performs the calculation:
  ```solidity
  uint256 unclaimedTitn = remainingTitnAfter1Year - initialTotalClaimable;
  ```
  However, `initialTotalClaimable` can be larger than `remainingTitnAfter1Year`. This happens if the total claimable TITN generated by users exceeds the physical TITN balance of the contract. There are two clear code paths that cause this:
  1. No cap is enforced on `tgt` deposits in `onTokenTransfer()`. If users deposit more than the expected `TGT_TO_EXCHANGE` TGT, `totalTitnClaimable` will naturally grow larger than the deposited `TITN_ARB` amount.
  2. The contract owner can withdraw TITN directly from the contract via the `withdraw()` function, reducing the actual TITN balance below the accumulated claimable TITN of users.
  
  In either case, `remainingTitnAfter1Year < initialTotalClaimable` will hold true, and the subtraction will revert due to arithmetic underflow.
- **Impact:** Once the 360-day mark passes, any attempt by eligible users to call `withdrawRemainingTitn()` will permanently revert, trapping all remaining TITN in the contract forever.

## 2. Griefing/DoS Vector: Freezing Arbitrary Addresses and Pools via LayerZero Bridges
- **Location:** `Titn.sol` : `_credit()` & `_validateTransfer()`
- **Mechanism:** When TITN is bridged across chains via LayerZero, the recipient is credited with tokens using the internal `_credit()` function, which permanently flags the address:
  ```solidity
  if (!isBridgedTokenHolder[_to]) {
      isBridgedTokenHolder[_to] = true;
  }
  ```
  In `_validateTransfer()`, transfers from any address marked as a `isBridgedTokenHolder` are immediately blocked as long as the global variable `isBridgedTokensTransferLocked` is set to `true`. Because anyone can initiate a cross-chain transfer of any amount to any recipient address, an attacker can bridge 1 wei of TITN via LayerZero to any target address (e.g., active wallets, decentralized exchange pools, multi-sigs, or staking agreements) on any destination chain. This forces `isBridgedTokenHolder[target] = true` and permanently freezes all transfers originating from that target.
- **Impact:** Attackers can cheaply and irreversibly freeze TITN transfers for any participant, contract, or liquidity pool in the ecosystem.

## 3. Whitelist Deadlock Preventing Multi-Contract Operations
- **Location:** `Titn.sol` : `_validateTransfer()`
- **Mechanism:** The transfer restrictions in `Titn` can only be bypassed on locked chains (such as Arbitrum) if either the sender or receiver is the `transferAllowedContract`:
  ```solidity
  from != transferAllowedContract &&
  to != transferAllowedContract
  ```
  However, `transferAllowedContract` is stored as a single `address` slot, allowing only one contract to be whitelisted at a time. This design creates an operational deadlock. For example, if `MergeTgt` is whitelisted so that users can claim and withdraw their TITN, no other contract (such as a Uniswap pool, a staking pool, or a governance vault) can be whitelisted. If the owner switches `transferAllowedContract` to a Uniswap pool to facilitate trading, all claims and owner withdrawals from `MergeTgt` will instantly fail and revert.
- **Impact:** The protocol cannot support multiple ecosystem contracts simultaneously, resulting in frozen assets, inability to claim, or a total breakdown of trading and staking utilities.

## 4. User Loss of Funds at the Exactly 360-Day Mark
- **Location:** `MergeTgt.sol` : `onTokenTransfer()` & `quoteTitn()`
- **Mechanism:** In `onTokenTransfer()`, the deadline check to accept deposits is:
  ```solidity
  if (block.timestamp - launchTime > 360 days) {
      revert MergeEnded();
  }
  ```
  If a deposit transaction occurs at exactly `block.timestamp - launchTime == 360 days`, this check does not revert and the TGT tokens are transferred into the contract. However, `quoteTitn()` evaluates the payout amount using:
  ```solidity
  else if (timeSinceLaunch < 360 days) { ... }
  else { titnAmount = 0; }
  ```
  At exactly 360 days, `timeSinceLaunch < 360 days` is FALSE, so `quoteTitn()` falls into the `else` branch, returning `0`.
- **Impact:** Users who transact exactly at the 360-day mark will have their TGT tokens permanently taken and locked in the contract, whilst receiving `0` claimable TITN tokens in return.
