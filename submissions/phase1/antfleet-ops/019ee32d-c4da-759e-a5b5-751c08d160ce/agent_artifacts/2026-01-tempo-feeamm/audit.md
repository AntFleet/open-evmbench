# Audit: 2026-01-tempo-feeamm

## Arbitrage to Drain Pool via Single-Sided Minting and Proportional Burning
- Location: `FeeAMM.sol` : `mint` and `burn`
- Mechanism: The contract uses a valuation discrepancy between single-sided LP addition and double-sided LP removal. LP minting only accepts `validatorToken` but values the pool's existing `userToken` holdings with a discount factor of `N / SCALE` (0.9985) in the denominator of the LP share calculation. In contrast, burning LP tokens returns both `userToken` and `validatorToken` proportionally to their current pool reserves without any discount factor. Because LP shares in the pool's discounted assets are purchased at an undervalued rate and withdrawn at full value, any single-sided deposit followed by an immediate withdrawal yields an instantaneous, risk-free profit.
- Impact: An attacker can exploit this design flaw to continuously extract funds from existing liquidity providers. By using a flash loan to deposit a large amount of `validatorToken` via `mint` and immediately executing `burn` in the same transaction, the attacker can siphon off the pool's assets and eventually drain the underlying reserves of both tokens.

## Reentrancy Vulnerability in LP Burning
- Location: `FeeAMM.sol` : `burn`
- Mechanism: The `burn` function violates the checks-effects-interactions pattern by transferring `userToken` and `validatorToken` to the recipient prior to updating the state variables. Specifically, the transfer calls `IERC20(userToken).transfer(to, amountUserToken)` and `IERC20(validatorToken).transfer(to, amountValidatorToken)` occur before updating the user's LP balances (`liquidityBalances[poolId][msg.sender]`), the `totalSupply[poolId]`, and the pool reserves. If either of the tokens are ERC-777 or have custom transfer hooks that hand control back to the recipient, the caller can execute a reentrant call back into the `burn` function.
- Impact: An attacker can repeatedly reenter the `burn` function using the same initial LP token balance. Because the contract does not decrement the caller's LP balance or the pool reserves until after the transfers are completed, the reentrant calls will successfully bypass the liquidity balance checks, allowing the attacker to multiply their withdrawals and drain all ERC20 tokens held by the AMM.

## Unchecked Integer Underflows and Overflows due to Missing SafeMath
- Location: `FeeAMM.sol` : `mint`, `burn`, and `executeFeeSwap`
- Mechanism: The contract specifies Solidity version range `>=0.7.6 <0.8.0` where arithmetic overflows and underflows do not automatically revert. However, the contract does not import or use OpenZeppelin's `SafeMath` library for several state variable modifications. For instance, in `mint()`, the additions `pool.reserveValidatorToken += uint128(amountValidatorToken)`, `totalSupply[poolId] += liquidity`, and `liquidityBalances[poolId][to] += liquidity` are not protected against overflow. Likewise, in `executeFeeSwap()`, `pool.reserveUserToken += uint128(amountIn)` can overflow and wrap around silently.
- Impact: Attackers or ordinary users can overflow state variables to corrupt the pool's accounting. For example, if a reserve wraps around to a small number, the true token balance of the contract becomes completely disconnected from the tracked state. This can lock LP withdrawals, cause permanent loss of funds, or allow users to artificially inflate their LP share balance to steal assets.
