# FeeAMM Security Audit

## Reentrancy in `burn` allows draining pool
- **Location:** `contracts/FeeAMM.sol` : `burn()`
- **Mechanism:** The `burn` function performs two external `IERC20.transfer` calls *before* updating any state (liquidity balance, total supply, and reserves). A malicious token (ERC777-style, or any token whose `transfer` invokes a hook) can re-enter `burn` during the first `transfer`. The re-entrant call will pass the `liquidityBalances >= liquidity` check because that balance has not yet been decremented, and it will compute the same `amountUserToken`/`amountValidatorToken` from unchanged reserves, then transfer them out and decrement state a second time. When control returns to the original frame, the second `transfer` (of `validatorToken`) will still pass (the attacker as `to` can re-deposit), and the outer call's final state updates will underflow (Solidity 0.7 has no built-in overflow protection), corrupting `liquidityBalances`, `totalSupply`, and the reserves to near-`uint256.max` values.
- **Impact:** An attacker controlling a token used in the pool (or controlling `to` with a hooked token) can drain the full balance of both reserves and permanently brick the pool's accounting, since subsequent `mint`/`burn`/`swap` operations will operate on corrupted invariants.

## Unchecked ERC20 return values
- **Location:** `contracts/FeeAMM.sol` : `executeFeeSwap`, `rebalanceSwap`, `mint`, `burn`
- **Mechanism:** Every token interaction (`transferFrom`, `transfer`) is called as a bare statement with no check on the boolean return value. Non-standard tokens (e.g., USDT on mainnet, or any token that returns `false` rather than reverting on failure) will silently leave the contract's internal accounting desynchronised from its actual token balances. In `executeFeeSwap` and `rebalanceSwap` this desynchronisation is immediate: the contract decrements `reserveValidatorToken`/`reserveUserToken` and increments the other side even when the corresponding transfer did not move any tokens.
- **Impact:** A user (or a hostile token) can trigger a swap where `transferFrom` returns `false`. The pool's reserves are updated as if the transfer succeeded, so the pool records owning tokens it never received. Subsequent honest users can be paid out from a pool that is actually undercollateralised, and LPs find their `burn` reverts because the calculated payout exceeds the real `IERC20.balanceOf`.

## `mint` never pulls the user-token side, breaking the initial invariant
- **Location:** `contracts/FeeAMM.sol` : `mint`
- **Mechanism:** The function signature only accepts `amountValidatorToken` and only calls `transferFrom` on `validatorToken`. `pool.reserveUserToken` is never incremented on initial deposit. The initial-liquidity branch nevertheless computes `liquidity = amountValidatorToken/2 - MIN_LIQUIDITY` and credits `MIN_LIQUIDITY` to the dead supply — i.e. it assumes the depositor contributed `amountValidatorToken/2` of *both* tokens. After a fresh `mint` the pool therefore holds `amountValidatorToken` validator tokens against `0` user tokens, while the LP holds liquidity tokens priced as if both sides were present. The follow-up branch (`product = N * reserveUserToken / SCALE`) then uses this always-zero `reserveUserToken` (until a user swap arrives) to compute LP shares, severely underpricing LP tokens relative to what the depositor actually paid.
- **Impact:** (1) `rebalanceSwap` is unsalvageable immediately after a fresh `mint` because `require(pool.reserveUserToken >= amountOut)` can never be satisfied, so a validator who needs to rebalance cannot do so. (2) LPs are silently diluted: the next `executeFeeSwap` from a user is the *only* mechanism that ever seeds `reserveUserToken`, and the LP's claim is computed against a denominator that omits the validator-token inventory the user-side flow is implicitly subsidising. (3) Any LP who burns before that first user swap receives exactly `0` user tokens, meaning their validator-token inventory is shared with no offsetting asset.

## No slippage / minimum-output parameters on swaps
- **Location:** `contracts/FeeAMM.sol` : `executeFeeSwap`, `rebalanceSwap`
- **Mechanism:** Neither swap function accepts `minAmountOut` (or `maxAmountIn` / `deadline`). Once a transaction is signed and in the mempool, any searcher can sandwich it: front-run with a swap that drains the validator-token side of the pool, then back-run after the victim's swap executes against the now-empty side. Because the formula is purely linear (`amountOut = amountIn * M / SCALE`, `amountIn = amountOut * N / SCALE + 1`) the post-sandwich rate is bounded only by `pool.reserveValidatorToken >= amountOut`, which the searcher controls.
- **Impact:** Honest users (validators and fee payers) can be made to receive far worse effective rates than the quoted 0.3% / 0.15% fee. Combined with the linear-pricing flaw below, a sandwicher can consistently extract the entire validator-token reserve of a pool from a single victim transaction.

## Linear pricing means the pool can be trivially drained to one side
- **Location:** `contracts/FeeAMM.sol` : `executeFeeSwap`, `rebalanceSwap`
- **Mechanism:** `executeFeeSwap` computes `amountOut = amountIn * M / SCALE` regardless of reserves; `rebalanceSwap` computes `amountIn = amountOut * N / SCALE + 1` regardless of reserves. There is no constant-product (or any reserve-dependent) invariant, so the pool never becomes "deeper" or "shallower" in price terms. An attacker with a sufficiently large user-token balance can iterate `executeFeeSwap` until `reserveValidatorToken == 0` (only bounded by the require on `reserveValidatorToken >= amountOut`), leaving the pool holding the attacker's full user-token input against zero validator tokens. The `mint` follow-up branch (using `product = N * reserveUserToken / SCALE`) will then credit the next LP as if `reserveValidatorToken` were the dominant side, but `reserveValidatorToken` is `0`.
- **Impact:** (1) The pool's validator-token side can be emptied in a finite number of swaps at the attacker's chosen pace, with the attacker's only cost being the linear 0.3% fee. (2) The two fee rates are asymmetric: a round-trip `executeFeeSwap` then `rebalanceSwap` of `X` user tokens yields less than `X` user tokens back (because of the `+1` in `rebalanceSwap`'s formula), so the system always leaks value to LPs only when user-token flow is one-directional; validators attempting to rebalance pay the `+1` rounding tax. (3) Combined with the missing user-token transfer in `mint`, an attacker can construct a position that is provably insolvent for existing LPs.

## `executeFeeSwap` and `rebalanceSwap` violate checks-effects-interactions
- **Location:** `contracts/FeeAMM.sol` : `executeFeeSwap`, `rebalanceSwap`
- **Mechanism:** In `executeFeeSwap` the order is `transferFrom(userToken)` → reserve updates → `transfer(validatorToken)`. In `rebalanceSwap` the order is reserve updates → `transferFrom(validatorToken)` → `transfer(userToken)`. `executeFeeSwap` in particular has a classic cross-function reentrancy shape: the first external call (`transferFrom`) is made before any state is written, so a malicious `userToken` whose `transferFrom` re-enters `executeFeeSwap` (or `mint`/`burn`) will observe stale reserves. The `rebalanceSwap` ordering is slightly better but still places an external `transferFrom` *after* it has already credited `reserveValidatorToken += amountIn` — a hook in `validatorToken` re-entering `burn` will see the inflated reserve and be able to withdraw against tokens that have not yet been collected.
- **Impact:** A hooked/ERC-777-style token deployed as either side of a pool lets an attacker multiply their effective withdrawal (as analysed in the `burn` reentrancy finding) or front-load a swap against reserves that have not yet been debited, enabling pool drainage at sub-fee cost.

## Unbounded zero-amount swap can grief-initialise arbitrary pool keys
- **Location:** `contracts/FeeAMM.sol` : `executeFeeSwap`
- **Mechanism:** `executeFeeSwap` does not require `amountIn > 0` and does not require the pool to be initialised. With `amountIn = 0` the function computes `amountOut = 0`, passes the `reserveValidatorToken >= amountOut` check (since `0 >= 0`), and proceeds to perform `transferFrom(0)` and `transfer(0)`. The first such call materialises a `Pool` struct in storage for an arbitrary `(userToken, validatorToken)` pair, and — depending on token behaviour — may also mark the contract as having a non-zero `balanceOf` from the zero-amount call.
- **Impact:** Anyone can pollute the `pools` mapping with arbitrary token pairs, inflating contract storage and (for tokens whose `balanceOf` becomes non-zero after a zero-value `transferFrom`) desynchronising off-chain indexers that key off `pools[poolId] != 0` to detect a real pool.
