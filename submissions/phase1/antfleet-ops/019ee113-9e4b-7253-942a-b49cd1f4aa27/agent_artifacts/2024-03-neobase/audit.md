# Audit: 2024-03-neobase

# Security Audit Report

## Gauge removal is broken for non-zero-weight gauges
- Location: `src/GaugeController.sol` : `remove_gauge()` / `_remove_gauge_weight()`
- Mechanism: In `remove_gauge()`, the gauge's type slot is set to zero (`gauge_types_[_gauge] = 0`) *before* `_remove_gauge_weight()` is called. Inside `_remove_gauge_weight()`, the gauge type is recomputed as `gauge_types_[_gauge] - 1`, which becomes `-1` (an `int128`). When this is implicitly widened to `uint256` for mapping access, `_get_sum(-1)` reads from storage slot `type(uint256).max`, returning 0. The subsequent line `uint256 new_sum = old_sum_bias - old_weight_bias;` then underflows (Solidity 0.8 checked arithmetic) whenever the gauge ever received any votes, reverting the entire transaction. The same flaw exists in the standalone `remove_gauge_weight()` function for the same reason.
- Impact: Governance cannot remove or zero-out any gauge that has ever accumulated vote weight. The protocol is stuck with gauges that may need to be retired (e.g., compromised, deprecated, or exploited pools), and a governance key compromise gives the attacker an immutable backdoor into the reward distribution.

## Voting power is permanently trapped in removed gauges
- Location: `src/GaugeController.sol` : `vote_for_gauge_weights()`
- Mechanism: The first require explicitly permits voting with weight 0 on non-gauges (`_user_weight == 0 || gauge_types_[_gauge_addr] != 0`), and the inline comment states "We allow withdrawing voting power from invalid (removed) gauges". However, the subsequent check `require(gauge_type >= 0, "Gauge not added")` runs unconditionally after `gauge_type = gauge_types_[_gauge_addr] - 1`, which is `-1` for any address that is not a registered gauge. So even passing `_user_weight = 0` reverts for removed gauges. All the per-user bookkeeping that follows (`vote_user_slopes`, `vote_user_power`, slope-changes for both the gauge and the type sum) is also keyed by the (now invalid) `-1` index.
- Impact: Users can never reduce or reset votes on gauges that governance has removed. Their allocated voting power (and the corresponding `vote_user_power` budget of up to 10 000 bps) is permanently consumed, preventing them from reallocating to active gauges. In effect, governance can grief any user's remaining voting power by removing a gauge they voted on.

## LendingLedger address redirection creates fragmented user accounting
- Location: `src/LendingLedger.sol` : `sync_ledger()`
- Mechanism: `sync_ledger` rewrites `lendingMarket` based on `liquidityGauges[msg.sender]`. If `msg.sender` is the underlying market, `lendingMarket` is rewritten to the *gauge* address and all state is written under `userInfo[gauge][user]`, `lendingMarketTotalBalance[gauge]`, and `marketInfo[gauge]`. If `msg.sender` is the gauge (via `_afterTokenTransfer`), `lendingMarket` is rewritten to the underlying *market* address and the same fields are written under the market key. The reverse mapping in `whiteListLendingMarket` (`liquidityGauges[gauge] = market`, `liquidityGauges[market] = gauge`) is symmetric, so whichever side initiates the call dictates the keying. `claim(_market)` does not perform any redirection, so a user must know which key their balance is stored under to claim.
- Impact: The same user can have their ledger state split across two different keys depending on whether they interacted with the market directly or via the gauge. This fragments `userInfo`, `lendingMarketTotalBalance`, and `accCantoPerShare` accumulation: rewards can be calculated against the wrong `marketSupply`, the same deposit can be double-counted or zero-counted across the two ledgers, and `claim` may revert (or silently pay out zero) because `userInfo[_market][msg.sender]` is empty.

## Secondary rewards are accrued but can never be withdrawn
- Location: `src/LendingLedger.sol` : `sync_ledger()` / `claim()`
- Mechanism: `sync_ledger` maintains `user.secRewardDebt` and `market.secRewardsPerShare`, and `update_market` correctly accrues `secRewardsPerShare` per block. The `claim()` function, however, only computes and pays out the primary `cantoToSend`; it never reads `secRewardsPerShare`, never updates `user.secRewardDebt`, and never transfers any secondary reward token. No other function in the contract pays out secondary rewards.
- Impact: Every deposit/withdraw correctly tracks a secondary-reward entitlement, but the entitlement is permanently locked. There is no way for a user, governance, or any other role to extract the secondary reward balance. If a secondary reward token is ever configured (e.g., via a future hook or external distributor), the accounting will silently accumulate value that is unrecoverable.

## Historical gauge weight is replaced with the current weight for reward accounting
- Location: `src/LendingLedger.sol` : `update_market()`
- Mechanism: Inside the per-epoch loop, `epochTime` is computed once from `block.number` (the current block), not from `i` (the block actually being processed). `gauge_relative_weight_write(_market, epochTime)` is then called with that current-time value, so the gauge relative weight used for the entire backfilled range is the weight at the *current* block, not the weight at the historical epoch. Because `gauge_relative_weight_write` performs a write-checkpoint via `_get_weight` and `_get_total`, this also drags the GaugeController's `time_weight` / `time_total` forward.
- Impact: Reward distribution is no longer "vote-escrow": ve-token holders can change or remove their votes immediately before any user triggers `update_market` and capture a full epoch's rewards at the new weight, while the historical weights that were supposed to govern the epoch are ignored. This is a direct deviation from the Curve vote-escrow model the comment block describes and a manipulable oracle for emissions.

## `uint128` reward accumulators are destined to DoS the market
- Location: `src/LendingLedger.sol` : `update_market()`
- Mechanism: `MarketInfo.accCantoPerShare` and `secRewardsPerShare` are both `uint128`. The accumulation uses `market.accCantoPerShare += uint128((blockDelta * cantoPerBlock[epoch] * gauge_relative_weight_write(...)) / marketSupply);`. With `cantoPerBlock` and `gauge_relative_weight_write` each potentially returning `uint256` values and `blockDelta` growing with the number of uncheckpointed blocks, the per-iteration delta can easily exceed `uint128` in a single step (or wrap after enough epochs). Solidity 0.8 will revert on the cast/increment, and because the increment is inside the unconditional `while (i < block.number)` loop, the entire `update_market` call reverts.
- Impact: Once `accCantoPerShare` approaches `type(uint128).max`, every `sync_ledger` and `claim` for that market reverts. All deposits, withdrawals, and reward claims for that market are bricked. This is a slow-motion denial of service that hits every market eventually under normal operation.

## `update_market` is an unbounded, attacker-controllable gas bomb
- Location: `src/LendingLedger.sol` : `update_market()`
- Mechanism: `update_market` is `public` (no access control) and iterates `while (i < block.number)` with no cap other than the implicit uint256 limit. A market that legitimately has not called `sync_ledger` (or has been de-whitelisted and re-whitelisted, which resets `lastRewardBlock`) can accumulate an arbitrary block delta. The loop body also performs an external call to `gaugeController.gauge_relative_weight_write`, which itself runs two 500-iteration checkpoints. Anyone (including a griefing attacker) can call `update_market` for a whitelisted market and force it to consume all the gas in the block.
- Impact: A malicious actor can permanently DoS the lending market's reward accounting by triggering `update_market` when the block delta is large, reverting every subsequent `sync_ledger`/`claim` for that market. Even without malicious intent, an extended period of market inactivity (e.g., during the de-whitelist/re-whitelist reset path) can push the loop past the block gas limit and brick the market.

## `power_used` underflow guard is broken and meaningless
- Location: `src/GaugeController.sol` : `vote_for_gauge_weights()`
- Mechanism: `uint256 power_used = vote_user_power[msg.sender]; power_used = power_used + new_slope.power - old_slope.power;` performs an unchecked-feeling subtraction on a `uint256`; the subsequent `require(power_used >= 0, ...)` is a no-op (it is always true for an unsigned value). The only real guard is `power_used <= 10_000`. If `old_slope.power > power_used + new_slope.power` (e.g., a user's previously stored power is somehow larger than the recomputed value due to prior state corruption, storage aliasing, or a future accounting change), the subtraction reverts under checked arithmetic and locks the user out of voting entirely.
- Impact: Any inconsistency between the stored `vote_user_slopes[user][gauge].power` and the actual accumulated `vote_user_power[user]` permanently bricks the affected user's voting rights. The intended "soft" validation is replaced by an underflow revert that, combined with finding #2, can leave users with no path to recover their voting budget.

## Out-of-bounds gauge-type indexing in view functions
- Location: `src/GaugeController.sol` : `_gauge_relative_weight()` and `gauge_relative_weight()`
- Mechanism: The view function computes `int128 gauge_type = gauge_types_[_gauge] - 1` with no zero-check; for any non-registered address this is `-1`. The expression `points_type_weight[gauge_type][t]` then performs an implicit `int128 -> uint256` conversion, reading storage slot `type(uint256).max`. Because `_type_weight` from that slot can be non-zero, and `points_weight[_gauge][t].bias` is also a fresh storage cell, the function can return a non-zero, attacker-chosen-looking value for a gauge that does not exist.
- Impact: Off-chain consumers and other on-chain contracts that call `gauge_relative_weight` (e.g., integrators, the `update_market` path) can be fed garbage ratios for arbitrary addresses. There is no input validation, and a malicious gauge could be created whose storage layout collides with the high slot to deliberately influence reported weights.

## `liquidityGauges` reverse mapping breaks market identity invariant
- Location: `src/LendingLedger.sol` : `whiteListLendingMarket()` / `sync_ledger()`
- Mechanism: `whiteListLendingMarket` populates `liquidityGauges[_market] = gauge` and `liquidityGauges[gauge] = market`. `sync_ledger` then uses this map to "redirect" between market and gauge. But `lendingMarketTotalBalance`, `marketInfo`, and `userInfo` are all keyed by the *redirected* address, while `claim(_market)` is keyed by the address the user supplies. A gauge's market can change semantics: de-whitelisting the market leaves `liquidityGauges[gauge] = market` pointing to an address that `update_market` will now reject as "Market not whitelisted", and the gauge's `_afterTokenTransfer` will revert on every transfer, permanently locking user balances inside the ERC20 gauge.
- Impact: De-whitelisting a market that owns a gauge bricks the gauge contract. All gauge transfers revert, users can neither deposit via the gauge nor exit, and the underlying tokens sit trapped in the gauge. This is a denial-of-service of user funds triggered by a routine governance action.

## Receive hook silently increases payout liability
- Location: `src/LendingLedger.sol` : `receive()` and `claim()`
- Mechanism: The contract accepts arbitrary native `CANTO` via `receive()` with no event, no accounting, and no way to refund or attribute the funds. `claim` pays out `cantoToSend` derived purely from `accCantoPerShare` math and ignores the actual contract balance. If the contract's CANTO balance ever falls below the total pending `cantoToSend` across all users (e.g., because governance set a large `cantoPerBlock` rate, or because donations were sent and then drained via a separate bug), `claim` reverts for all users.
- Impact: A griefing attacker can `selfdestruct`-force CANTO into the contract, temporarily masking an under-funded payout, or conversely can drain accidentally-donated CANTO via a separate bug. More importantly, the protocol's payout is not actually backed by a tracked balance, so an under-collateralized `cantoPerBlock` rate silently breaks all claims with no on-chain indicator of the shortfall.
