# Audit: 2024-03-abracadabra-money

# Security Audit Report

## Broken MagicLpAggregator Oracle
- **Location**: `src/oracles/aggregators/MagicLpAggregator.sol` : `_getReserves` / `latestAnswer`
- **Mechanism**: The `_getReserves` function declares local variables `baseReserve` and `quoteReserve` but never assigns them to the unnamed return values, so the function always returns `(0, 0)`. As a result, `latestAnswer` always returns `0` because `minAnswer * (0 + 0) / pair.totalSupply() == 0`. More critically, `MagicLpAggregator` only implements `IAggregator` (which has no `peek`/`get`/`peekSpot` methods) and does **not** implement the `IOracle` interface used by Cauldrons. If a `ProxyOracle` is pointed at this implementation, calls to `peek`, `peekSpot`, or `get` will revert, making any Cauldron using it inoperable; if the contract is ever wired directly into a Cauldron (e.g. via a custom adapter that does not exist in the codebase), the returned price of 0 would make every borrower permanently solvent (since `borrowValue * 0 == 0`) while simultaneously causing divisions by zero in `getUserPositionInfo` / `getLiquidationCollateralAndBorrowAmount`.
- **Impact**: Oracles priced via MagicLP will return 0 (or revert), enabling unbounded uncollateralized borrowing on any Cauldron that successfully resolves the price, and bricking any Cauldron that resolves it through standard `IOracle` methods.

## BlastOnboarding Proxy Implementation Is Arbitrarily Upgradeable
- **Location**: `src/blast/BlastOnboarding.sol` : `_implementation` / `setBootstrapper`
- **Mechanism**: The `Proxy._implementation()` returns `address(bootstrapper)`, and `setBootstrapper` is a plain `onlyOwner` setter with no timelock, no zero-address check, and no restriction on what contract is set. After users have deposited into `BlastOnboarding` and the state is `Closed` (funds are sitting in the proxy), the owner can call `setBootstrapper(maliciousContract)`. Because the proxy uses `delegatecall`, the malicious contract's code runs in the proxy's storage context and can arbitrarily overwrite `totals`, `balances`, `supportedTokens`, etc., then drain the held MIM/USDB. There is no `init`/`upgrade` function on the implementation side that validates the new layout, so a storage-colliding or simply destructive contract is sufficient.
- **Impact**: Single point of failure — the owner (or anyone who compromises the owner key) can steal 100% of the funds held in the BlastOnboarding proxy at any time, including user deposits between `close()` and `claim()`.

## Unrevoked Max Approvals to Router in BlastOnboarding
- **Location**: `src/blast/BlastOnboardingBoot.sol` : `bootstrap`
- **Mechanism**: `bootstrap()` calls `MIM.safeApprove(address(router), type(uint256).max)` and `USDB.safeApprove(address(router), type(uint256).max)` and never revokes or reduces these allowances. The router is immutable on the `Factory` it points to, but the approval is unbounded and persistent for the lifetime of the Onboarding contract. Any future vulnerability in the router, or any future change to the router address, leaves the Onboarding contract permanently exposed: the approved spender can pull `type(uint256).max` of MIM and USDB at any time.
- **Impact**: Permanent, unlimited allowance to a third party over user-deposited funds in the Onboarding contract; a single router bug or key compromise drains the entire balance.

## `DegenBox.transferMultiple` Skips Critical Validation
- **Location**: `src/DegenBox.sol` : `transferMultiple`
- **Mechanism**: The function only checks `tos[0] != address(0)`, ignoring all other entries, and does not verify that `tos.length == shares.length`. Recipients at indices `i > 0` can be `address(0)`, and the corresponding shares are silently added to the zero address (effectively burned from the sender). The sender's balance is reduced by `totalAmount` at the end, so the loss is real. Additionally, the loop does an `add` to each `to` before the `sub` from `from`; a recipient contract re-entering the BentoBox through the same `from` (e.g. via a `transfer` that triggers another `transferMultiple`) can desynchronize the accounting, although in practice the `from` balance check at the end reverts any over-spend.
- **Impact**: Users who pass a `tos` array containing an unintended `address(0)` element (e.g. through a buggy UI) silently burn the corresponding shares. There is no length check, so a UI bug that supplies mismatched arrays either reverts (wasting gas and confusing the user) or silently sends the wrong amounts.

## `LockingMultiRewards.notifyRewardAmount` Compares Incompatible Units
- **Location**: `src/staking/LockingMultiRewards.sol` : `notifyRewardAmount`
- **Mechanism**: The guard `if (amount < _remainingRewardTime) revert ErrNotEnoughReward();` compares the reward token amount (in token wei) to a duration in seconds. These quantities have no fixed numerical relationship. For any reward token with fewer than ~18 decimals, even modest rewards will satisfy the comparison as intended, but for tokens with 18 decimals a reward of 1 token (`1e18`) compared against 7 days (`604800`) passes, while a reward of 1 wei fails — the check is effectively nonsensical and is "passing" for the wrong reason. If the token has very small decimals (e.g. a token with 0 decimals, or the comparison is done in raw units that happen to be larger than the duration), small legitimate rewards will be rejected.
- **Impact**: Reward distributions for low-decimal tokens (or very small amounts of any token) will revert, causing the staking contract's reward stream to halt. This is a denial-of-service against the reward mechanism, not a direct fund loss, but it locks all stakers out of pending rewards.

## `DegenBox` Strategy Balance Used in Flash-Loan Solvency Check
- **Location**: `src/DegenBox.sol` : `flashLoan` / `batchFlashLoan` / `_tokenBalanceOf`
- **Mechanism**: `_tokenBalanceOf` returns `token.balanceOf(address(this)) + strategyData[token].balance`. The `strategyData[token].balance` is the BentoBox's *belief* about the strategy's balance, updated only when `harvest` is called. If a strategy is exploited, drained, or simply reports a stale value, the flash-loan repayment check `_tokenBalanceOf(token) >= totals[token].addElastic(fee)` will pass even though the actual on-chain funds are insufficient. A borrower could then take a flash loan, leave only enough tokens to satisfy the (inflated) `strategyData` balance, and walk away with the difference once the strategy is eventually `harvest()`ed (at which point the loss is socialized across depositors via a `subElastic`).
- **Impact**: Combined with a compromised or buggy strategy, flash loans can be used to extract the gap between the strategy's reported balance and its real on-chain holdings, with the loss ultimately borne by all share holders via a downward share-price adjustment.
