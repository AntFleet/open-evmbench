# Audit: 2024-07-benddao

# Security Audit Findings

## Global pause mechanism is non-functional
- **Location:** `src/modules/Configurator.sol` : `setGlobalPause`; `src/base/Base.sol` : `whenNotPaused`
- **Mechanism:** The `setGlobalPause` function on `Configurator` calls `_pause()`/`_unpause()` on the `Configurator` contract's own `Pausable` storage. Every other module (e.g., `BVault`, `CrossLending`, `CrossLiquidation`, `IsolateLending`, `IsolateLiquidation`, `FlashLoan`, `Yield`) inherits `Pausable` separately and maintains its own `_paused` storage slot, but none of these modules expose a function to flip that flag. As a result, `whenNotPaused` in every lending/liquidation/yield function is always `false` (i.e. unpaused) regardless of the "global pause" state. An emergency admin invoking `setGlobalPause(true)` only pauses `Configurator`; user-facing actions — supply, borrow, withdraw, liquidate, flash loan, yield — remain executable.
- **Impact:** The protocol cannot be paused in an emergency (oracle hack, upgrade bug, exploit in progress). The advertised circuit-breaker is dead code.

## Lending modules are publicly callable with attacker-controlled `msgSender`
- **Location:** `src/base/BaseModule.sol` : `unpackTrailingParamMsgSender`; `src/modules/BVault.sol`, `src/modules/CrossLending.sol`, `src/modules/CrossLiquidation.sol`, `src/modules/IsolateLending.sol`, `src/modules/IsolateLiquidation.sol`, `src/modules/Yield.sol` (all `public` functions); `src/base/Proxy.sol` : `fallback`
- **Mechanism:** The Proxy/delegatecall pattern appends the original caller to calldata, and every module function reconstructs the sender with `unpackTrailingParamMsgSender()`. However, the modules are themselves deployed as standalone contracts with `public` functions — there is no `require(msg.sender == proxy)` check. An attacker calls a module function directly and crafts the trailing 20 bytes to be any address (e.g., a victim). The module then treats that arbitrary address as the legitimate sender for all authorization checks (`validateSenderApproved`, `tokenData.owner == msgSender`, `ERC721TokenData.owner == msgSender`, `stakeDatas[nft][tokenId]` ownership, etc.). For example, `BVault.setApprovalForAll` lets the attacker write `poolData.accountLookup[victim].operatorApprovals[asset][attacker] = true` for any victim/asset. With that approval, `CrossLending.crossBorrowERC20` can be called directly with `msgSender=attacker`, `onBehalf=victim`, `receiver=attacker`; the victim's collateral funds the borrow and the proceeds go to the attacker.
- **Impact:** Complete bypass of access control. An attacker can (a) grant themselves operator approval on any account, (b) borrow/withdraw/liquidate on behalf of any user, (c) delegate/stake any user's NFT collateral. This is full account takeover and direct theft of pool liquidity backed by victim collateral.

## `YieldBorrowERC20` cap is calculated against the borrow index, not the supply index
- **Location:** `src/libraries/logic/YieldLogic.sol` : `executeYieldBorrowERC20`
- **Mechanism:** The function computes the asset-level cap as
  `vars.totalSupply = VaultLogic.erc20GetTotalCrossSupply(assetData, groupData.borrowIndex);` and then enforces `(totalBorrow + amount) <= totalSupply * yieldCap`. `erc20GetTotalCrossSupply` simply multiplies the scaled supply by whatever index is passed. Supply and borrow indices diverge over time (the supply rate is a fraction of the borrow rate, and utilization < 1), so `borrowIndex` is normally larger than `supplyIndex`. Using the borrow index inflates `totalSupply` and therefore the effective cap, allowing the uncollateralized yield borrows to exceed the intended ceiling. The staker-level cap is checked identically and suffers the same inflation.
- **Impact:** The "yield" feature is designed to lend out a bounded slice of supply without collateral. Because the cap is computed against the wrong (larger) index, a whitelisted staker can drain a multiple of the configured `yieldCap` from the pool, leaving insufficient liquidity for ordinary suppliers.

## Cross-module reentrancy (per-contract reentrancy lock)
- **Location:** `src/base/Base.sol` : `nonReentrant`; `src/base/Storage.sol` : `reentrancyLock`; all `public` external functions in `src/modules/*.sol`
- **Mechanism:** `reentrancyLock` is declared `uint internal` in the `Storage` abstract contract. Every module (`BVault`, `CrossLending`, `CrossLiquidation`, `IsolateLending`, `IsolateLiquidation`, `FlashLoan`, `Yield`, etc.) inherits `Storage` independently, so each module has its own copy of the lock. The `nonReentrant` modifier only blocks re-entry into the *same* module. A malicious receiver (or a token with a hook) that re-enters via a *different* module bypasses the guard. Library logic in `VaultLogic` updates accounting state and performs external calls (e.g., `erc20TransferOutLiquidity` does `availableLiquidity -= amount; IERC20.safeTransfer(to, amount);`), and `PoolManager.dispatch` is a reentrantOK target reachable from any contract.
- **Impact:** Cross-function/cross-module reentrancy during supply, withdraw, borrow, repay, liquidation, and yield operations. Combined with the direct-call access-control bypass above, an attacker can interleave calls (e.g., liquidate-then-borrow-then-deposit) to manipulate mid-operation state such as `availableLiquidity`, `accruedFee`, scaled supply/borrow, and `accountLookup` flags.

## `VaultLogic` ERC20/ERC721 transfers update state before the external call
- **Location:** `src/libraries/logic/VaultLogic.sol` : `erc20TransferInLiquidity`, `erc20TransferOutLiquidity`, `erc20TransferInBidAmount`, `erc20TransferOutBidAmount`, `erc20TransferBetweenWallets`, `erc721TransferOutLiquidity`, `erc721TransferInLiquidity`
- **Mechanism:** In every transfer helper the internal accounting (`availableLiquidity += amount`, `totalBidAmout += amount`, scaled balances, etc.) is mutated *before* the external `safeTransfer`/`safeTransferFrom`. The post-call `balanceOf` check verifies the pool's token balance, but the protocol's internal `availableLiquidity` has already been credited/debited. For an ERC-777-like underlying (or any future asset the governance whitelist allows with a callback) the token's `tokensReceived` hook runs with the pool's state already updated, and combined with the cross-module reentrancy above the hook can borrow, withdraw collateral, or re-deposit against the inflated liquidity. `erc20TransferOutLiquidity` is particularly exposed because `availableLiquidity` is decremented before the external call, yet the pool's actual ERC20 balance is not yet reduced, creating a window for double-spend.
- **Impact:** On any whitelisted token with a transfer hook (or via proxy-mediated reentrancy) an attacker can drain the pool by repeatedly borrowing/withdrawing against the temporarily-inflated `availableLiquidity`, or by capturing liquidation collateral that has already been "burned" from the victim's scaled balance.

## Yield-staking share accounting is vulnerable to first-depositor inflation
- **Location:** `src/yield/YieldStakingBase.sol` : `_stake`, `_convertToYieldSharesWithTotalYield`, `convertToShares` (in `src/libraries/math/ShareUtils.sol`)
- **Mechanism:** Shares are minted with `assets.mulDiv(totalShares + 1, totalAssets + 1, rounding)` (the OpenZeppelin "virtual shares" pattern). The virtual offset is only `+1`, which protects against the trivial zero-deposit attack but not against the standard inflation attack when the first staker deposits a small amount and then donates a large amount of the yield-bearing token directly to the yield account. After such a donation, the rounding-down share calculation gives the attacker nearly all existing shares while subsequent honest stakers receive only `assets * 1 / (donation + 1)` shares, which rounds to 0 for any normal-sized deposit. The attacker then redeems the bulk of the account's yield at the victim's expense. The same pattern repeats in `_repay` and `_unstake` via `convertToYieldAssets`/`convertToYieldShares` (also `totalShares + 1 / totalAssets + 1`).
- **Impact:** The first staker (or anyone who can force a tiny initial deposit, e.g. via the bot admin path) can grief or steal the yield of every subsequent staker, since the rounding path lets them capture a disproportionate share of the protocol-minted yield token.

## Unstaking amount is included in `getAccountTotalUnstakedYield`, diluting new stakers
- **Location:** `src/yield/YieldStakingBase.sol` : `getAccountTotalUnstakedYield`, `_unstake`, `getProtocolTokenAmountInUnderlyingAsset`
- **Mechanism:** In `_unstake` the user's `sd.yieldShare` is set to 0 and `accountYieldShares[account]` is decremented, but the underlying yield-bearing token (eETH/stETH) is not transferred out of the yield account — it is locked in the protocol's withdrawal NFT until `protocolClaimWithdraw` is called. `getAccountTotalUnstakedYield(account)` returns the raw token balance (e.g., `eETH.balanceOf(account)`), which still includes the unstaking principal. Subsequent stakers therefore see an inflated `totalAssets`, receive fewer shares per unit deposited, and effectively subsidize the unstaking position. Because the inflation scales with the unstaking amount, a whale can park an arbitrarily large unstaking position and continuously extract value from new stakers.
- **Impact:** Systematic value leak from new stakers to positions that have already entered the unstake queue; the per-account share price is not conserved across the unstake boundary.

## `setAssetClassGroup` does not verify the group is enabled in the pool
- **Location:** `src/libraries/logic/ConfigureLogic.sol` : `executeSetAssetClassGroup`
- **Mechanism:** The function only checks the numeric range `GROUP_ID_LEND_MIN..GROUP_ID_LEND_MAX`. It does not verify `poolData.enabledGroups[classGroup] == true` (which `executeAddPoolGroup` is what sets). An admin can therefore point an asset's `classGroup` at a group that the pool has not enabled. Subsequent `IsolateLogic` borrows that key the debt group off `nftAssetData.classGroup` end up in a non-existent `groupLookup[classGroup]` whose `rateModel == 0`, causing `validateGroupBasic` to revert on every borrow and effectively bricking that asset.
- **Impact:** A misconfiguration (or a compromised pool admin) silently disables isolate-mode borrowing against an entire NFT collection, or — by subsequently enabling the group — can be used to change the rate model that backs existing positions without going through the normal add/remove pool-group flow.

## Chainlink price staleness is not validated against heartbeat
- **Location:** `src/PriceOracle.sol` : `getAssetPriceFromChainlink`
- **Mechanism:** The function only checks `updatedAt != 0` and `answeredInRound >= roundId`. It never compares `block.timestamp - updatedAt` against the aggregator's heartbeat (`aggregator.HEARTBEAT()` / `aggregator.timeouts`). A Chainlink feed that has not been updated for hours or days (e.g., low-liquidity asset, sequencer downtime, abandoned feed) still returns the last value. The same gap applies to `SDAIPriceAdapter.latestRoundData` / `latestAnswer`, which simply forwards the underlying DAI feed.
- **Impact:** Liquidations, borrow-capacity checks, and isolate-borrow health-factor checks can be performed against arbitrarily stale prices. During a market crash the protocol would still treat the pre-crash price as valid, allowing under-collateralized borrows and preventing liquidations of now-underwater positions.

## `Installer.installModules` allows overwriting the `INSTALLER` module id
- **Location:** `src/modules/Installer.sol` : `installModules`; `src/PoolManager.sol` : `constructor` (where `moduleLookup[MODULEID__INSTALLER]` is seeded)
- **Mechanism:** The constructor seeds `moduleLookup[1] = installerModule`. `installModules` does not reserve or guard any module ids; it blindly writes `moduleLookup[newModuleId] = moduleAddr` and re-points `trustedSenders[proxyAddr].moduleImpl` for any `newModuleId <= MAX_EXTERNAL_SINGLE_PROXY_MODULEID`. A pool admin can pass a malicious contract with `moduleId() == 1` (or any other reserved id, e.g. `MODULEID__CONFIGURATOR`) and have it replace the real installer/configurator. Because all module functions are `delegatecall`'d from `PoolManager`, the malicious module runs in the pool's storage context with full control over pool data, the ACL, the address provider, and all funds.
- **Impact:** A compromised pool admin (or an admin key obtained via the access-control bypass above) can install a fully privileged backdoor and drain the pool.

## `PoolManager.emergencyEtherTransfer` and `Proxy.emergencyEtherTransfer` have no amount cap
- **Location:** `src/PoolManager.sol` : `emergencyEtherTransfer`, `emergencyProxyEtherTransfer`; `src/base/Proxy.sol` : `emergencyEtherTransfer`
- **Mechanism:** Both functions transfer native ETH to an arbitrary `to` address using a low-level call with no upper bound and no reentrancy guard. `Proxy.emergencyEtherTransfer` only checks `msg.sender == creator`, so once the proxy is created the creator (PoolManager) is the sole guard, and PoolManager in turn only checks `onlyPoolAdmin`.
- **Impact:** A single pool admin key can sweep all ETH accidentally sent to the protocol or its proxies. Combined with the access-control bypass that lets a non-admin impersonate a pool admin, an attacker can exfiltrate every wei held by the pool and its module proxies in one call.

## `getIsolateLoanData` / `getIsolateAuctionData` are unbounded and DoS-prone when called on-chain
- **Location:** `src/libraries/logic/QueryLogic.sol` : `getIsolateLoanData`, `getIsolateAuctionData`; `src/modules/PoolLens.sol` : `getIsolateLoanDataList`, `getIsolateAuctionDataList`
- **Mechanism:** `getIsolateLoanData` returns `(address(0), 0, 0, 0, 0)` when `loanData.reserveAsset == address(0)`, and `getIsolateAuctionData` returns zeros when `loanData.loanStatus != LOAN_STATUS_AUCTION`. However, the per-asset / per-token state is stored in a mapping, and the surrounding `*List` functions accept arbitrary `nftAssets.length` arrays. An attacker can call these view functions from a contract (e.g., another module that does an on-chain query before a state change) with a huge array, consuming all gas in the calling transaction and reverting it. The same is true of the `for (uint i = 0; i < nftAssets.length; i++)` loops in `getIsolateCollateralDataList`, `getERC721TokenDataList`, and `getNftStakeDataList`.
- **Impact:** Griefing / DoS of any contract path that performs an on-chain lens query before executing a user action. This is exploitable in combination with the cross-module reentrancy to force victim transactions to revert at attacker-chosen times (e.g., during liquidation or deleveraging).
