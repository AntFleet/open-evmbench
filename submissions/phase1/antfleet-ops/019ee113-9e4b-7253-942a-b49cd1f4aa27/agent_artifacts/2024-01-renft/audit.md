# Audit: 2024-01-renft

# Security Audit Findings

## PAYEE orders are silently dropped — rental is never recorded
- Location: `smart-contracts/src/policies/Create.sol` : `_rentFromZone` and `_convertToItems`
- Mechanism: `_convertToItems` invokes `_processPayeeOrderConsideration` for `OrderType.PAYEE`, but that helper only *counts* rental vs. payment items — it never populates the `rentalItems` array. The `items` array that comes out of `_convertToItems` for a PAYEE order is therefore allocated but entirely zero-initialized. Worse, `_rentFromZone` only performs any work when `orderType.isBaseOrder() || orderType.isPayOrder()`; for PAYEE orders the entire rental-recording branch (including `STORE.addRentals`, `ESCRW.increaseDeposit`, hook invocation, and the `RentalOrderStarted` event) is skipped. Seaport nevertheless accepts the fulfilment and transfers assets because `validateOrder` still returns the magic value at the end. The matching stop-side is also missing: `Stop._validateRentalCanBeStoped` reverts with `Shared_OrderTypeNotSupported` for anything that isn't BASE/PAY, and `_settlePayment` only handles BASE/PAY. `OrderMetadata` and `OrderType` explicitly document PAYEE as a supported order type paired with PAY.
- Impact: A PAYEE rental transfers real assets through the conduit but never appears in `Storage.orders` or `Storage.rentedAssets`. The lender has no protocol mechanism to ever stop the rental, the rented tokens can never be reclaimed through the stop flow, escrow payments are never settled, and the matching `onStop` hooks will underflow/garble hook state (e.g. `RestrictedSelectorHook.activeRentals[safe]--` or `functionSelectorBitmap[identifier]` cleared for a rental that never registered). Effectively any user that fulfils a PAYEE order permanently takes the rented assets with no recourse for the lender.

## Hooks not invoked for PAYEE orders
- Location: `smart-contracts/src/policies/Create.sol` : `_rentFromZone` (the `_addHooks` call)
- Mechanism: `_addHooks(payload.metadata.hooks, seaportPayload.offer, payload.fulfillment.recipient)` lives strictly inside the `if (orderType.isBaseOrder() || orderType.isPayOrder())` branch, so any hook listed in a PAYEE order's metadata is silently ignored. The matching `Stop._removeHooks` always fires every hook that an order claims, regardless of order type. Hooks therefore see `onStop` (decrement / clean-up) without ever seeing the matching `onStart` (increment / set-up) for any PAYEE fulfilment.
- Impact: Hook state goes negative or otherwise inconsistent — for `RestrictedSelectorHook`, `activeRentals[safe]` is decremented for a rental that never incremented it (wrap-around to `type(uint256).max` in 0.8), and `functionSelectorBitmap[identifier]` is deleted for an identifier that was never stored. For `ERC1155RewardHook` / `ERC20RewardHook`, accrued rewards and `rentInfo[assetHash].lastRewardBlock` are calculated against an `amount` that was never deposited, distorting reward distribution. After enough PAYEE activity a hook's invariant checks can be made to revert, locking rental safes out of targets.

## `Kernel._migrateKernel` does not actually brick the old kernel
- Location: `smart-contracts/src/Kernel.sol` : `_migrateKernel`
- Mechanism: Migration iterates over modules and active policies and only calls `changeKernel(newKernel_)` plus `setActiveStatus(false)` on each policy. It never zeroes `executor`, `admin`, `getModuleForKeycode`, `getKeycodeForModule`, `modulePermissions`, `moduleDependents`, `getDependentIndex`, `activePolicies`, `hasRole`, or `isRole`. The `MigrateKernel` action then emits `ActionExecuted(MigrateKernel, ...)` and the function returns. The NatSpec explicitly claims "THIS ACTION WILL BRICK THIS KERNEL" but that is false — every state-mutating entry point on `Kernel` (`executeAction`, `grantRole`, `revokeRole`, plus the self-referential `ChangeExecutor` / `ChangeAdmin`) remains callable by whoever still holds the old `executor` / `admin`. Even policies that point at the new kernel will still answer `onlyKernel` checks against the *old* kernel address they were instantiated with if a delegated call is ever re-routed there (e.g. via the `Kernel` reference captured by a `Module` constructor for non-proxied modules, which is exactly how `Admin`, `Create`, `Stop`, `Guard`, `Factory` are deployed here).
- Impact: After a migration the old kernel is a live, fully privileged twin that can still install modules, activate policies, grant roles, change its own admin/executor, and even re-migrate. Any new policy the operator deploys post-migration could be activated on the wrong kernel by mistake, and a hostile or compromised old executor/admin retains the ability to enqueue actions that look legitimate on-chain. The misleading comment encourages operators to trust that the old kernel is dead.

## `Storage.addRentals` does not constrain `amount` to `1` for ERC721 rentals
- Location: `smart-contracts/src/modules/Storage.sol` : `addRentals`
- Mechanism: `addRentals` blindly does `rentedAssets[asset.rentalId] += asset.amount` for every `RentalAssetUpdate` it is handed. `Create._processBaseOrderOffer` and `_processPayOrderOffer` copy `offer.amount` straight through into the `Item` struct without asserting it equals `1` for `ItemType.ERC721`. ERC1155 items legitimately allow any amount, but the same storage path is used for both.
- Impact: If any caller (or future version of `Create`, or a hook-driven migration path) ever pushes an ERC721 `RentalAssetUpdate` with `amount > 1`, `rentedAssets[rentalId]` is incremented past `1` and `Storage.removeRentals` can never decrement it back to zero (Solidity 0.8 underflow revert). The corresponding `RentalOrder` is then permanently un-stoppable from the protocol's perspective — the lender can never reclaim the NFT. Seaport currently rejects this at order construction, but the storage layer itself is the last line of defence and is missing.
