# Audit: 2024-04-noya

# Security Audit Report

## BalancerFlashLoan: Broken token return mechanism makes flash loans inoperable
- Location: `contracts/connectors/BalancerFlashLoan.sol` : `receiveFlashLoan`
- Mechanism: In the final loop of `receiveFlashLoan`, the contract calls `BaseConnector(receiver).sendTokensToTrustedAddress(...)` to pull tokens back from the receiver connector. The implementation in `BaseConnector` checks `if (registry.isAnActiveConnector(vaultId, msg.sender))` before transferring. The `msg.sender` in that call is the `BalancerFlashLoan` contract, which is never registered as a connector anywhere in the system (neither in `addVault` nor in `addConnector`). Consequently the check always fails, the function returns `0` and no tokens are transferred. The subsequent `tokens[i].safeTransfer(msg.sender, amounts[i])` to the Balancer vault then reverts because the contract holds no tokens. The flash loan is unusable in its current form.
- Impact: The flash-loan functionality is completely broken; any attempt to use it will revert. If the design intent was to use flash loans for liquidations, arbitrage, or collateral swaps, those flows are unavailable.

## BalancerFlashLoan: Unrestricted arbitrary external execution and missing reentrancy guard
- Location: `contracts/connectors/BalancerFlashLoan.sol` : `receiveFlashLoan`
- Mechanism: The function iterates over `destinationConnector` and executes `destinationConnector[i].call{value:0, gas:gas[i]}(callingData[i])` for each entry. Neither `destinationConnector` nor `callingData` is restricted to whitelisted targets or function selectors — any contract and any calldata can be invoked, and the caller can specify per-call gas. Furthermore `receiveFlashLoan` itself lacks a `nonReentrant` modifier (only `makeFlashLoan` has it). A malicious or compromised keeper can therefore use the vault's flash-loaned balances to (a) call arbitrary external contracts using the loaned funds, (b) re-enter the `AccountingManager`, connectors, or oracles during the callback window, and (c) consume arbitrary amounts of gas in nested sub-calls, enabling complex cross-contract manipulation that would not be possible with the per-function `nonReentrant` guards inside the AccountingManager (those guards do not extend to the Balancer vault's reentrant callback).
- Impact: A compromised keeper can drain the loaned funds, manipulate the vault's positions, grief other users, or exploit cross-contract interactions enabled by arbitrary external calls. Even a non-malicious keeper can accidentally cause reentrancy through arbitrary calls, breaking assumptions in the AccountingManager and connectors.

## DolomiteConnector: Borrow position opened with `BalanceCheckFlag.None`
- Location: `contracts/connectors/Dolomite.sol` : `openBorrowPosition`
- Mechanism: `borrowPositionProxy.openBorrowPosition(0, accountId, marketId, _amountWei, AccountBalanceHelper.BalanceCheckFlag.None)` is invoked with `None` as the balance check flag, meaning neither the source account (0) nor the destination borrow account is checked for solvency/balance after the transfer. No subsequent health-factor or collateral check is performed in the function (unlike `AaveConnector.borrow`, which calls `getUserAccountData`). A manager — or a compromised manager key — can open a borrow account and move arbitrary amounts of the base token out of the default account into a borrow account that is already, or immediately becomes, undercollateralized.
- Impact: The vault can be placed into a state where a Dolomite borrow position is instantly liquidatable, allowing external liquidators to seize the vault's collateral at a discount. The lack of a post-condition health-factor check means there is no safeguard against this within the function.

## FraxConnector: Registry position removed before the on-chain withdrawal
- Location: `contracts/connectors/FraxConnector.sol` : `withdraw`
- Mechanism: The function computes `currentCollateral` and, if `withdrawAmount == currentCollateral`, immediately calls `registry.updateHoldingPosition(..., true)` to remove the position from the registry before invoking `pool.removeCollateral(withdrawAmount, address(this))`. If the `removeCollateral` call reverts (for example because the Frax pair's health-factor check fails, the pool is paused, or a token transfer returns less than expected), the whole transaction reverts and the registry is rolled back, so there is no permanent inconsistency from the revert. However, if the call succeeds but the actual transferred amount is less than `withdrawAmount` (e.g. fees, slippage, or rounding), the registry already records the position as removed while the connector may still hold residual collateral tracked off-ledger. The TVL helper in `AccountingManager` will not see that residual collateral, leading to a TVL/profit under-count.
- Impact: A mismatch between the registry's view of the position and the Frax pair's actual state can be exploited to make the vault appear less valuable than it is, suppressing performance-fee accounting and potentially allowing a later manager action to redeem/manipulate the residual collateral in an untracked way.

## PendleConnector: `decreasePosition` redeems SY with a minimum output of 1 wei
- Location: `contracts/connectors/PendleConnector.sol` : `decreasePosition`
- Mechanism: The function performs `IPStandardizedYield(address(SY)).redeem(address(this), _amount, _underlyingToken, 1, true)`. The `1` is the `minTokenOut` parameter, meaning the contract is willing to accept as little as 1 wei of the underlying token in exchange for `_amount` of SY. If the SY token has depegged, the underlying market is illiquid, or the SY/underlying pool is manipulated during the same block, an attacker (or a buggy keeper) can burn SY for essentially nothing, while the vault's accounting still credits the full SY amount. Because the position is only removed when `closePosition && isMarketEmpty(market)`, an attacker can repeatedly call this to extract value from the position.
- Impact: A manager key (or a keeper interacting with a malicious market) can drain the value of the SY position, with the position only being closed when the market is fully emptied, by which time the value is gone. The 1-wei minimum defeats the purpose of a slippage check.

## PendleConnector: Missing `nonReentrant` on swap functions
- Location: `contracts/connectors/PendleConnector.sol` : `swapYTForSY`, `swapYTForPT`
- Mechanism: Both `swapYTForSY` and `swapYTForPT` make external calls to `pendleRouter.swapExactYtForSy` / `pendleRouter.swapExactYtForPt` but lack the `nonReentrant` modifier that is present on the other functions in the contract. The Pendle router is an external, untrusted contract that can call back into the connector or the `AccountingManager` during the swap. Without reentrancy protection, a malicious Pendle router (or a router that is itself reentered through a third party) could re-enter the connector and double-spend approvals, double-burn YT, or manipulate the position state observed by `_getPositionTVL`.
- Impact: Reentrancy through the Pendle router can lead to loss of YT, double-counting in TVL, or manipulation of the position's recorded state.

## AaveConnector: `repayWithCollateral` lacks reentrancy guard
- Location: `contracts/connectors/AaveConnector.sol` : `repayWithCollateral`
- Mechanism: `repayWithCollateral` calls `IPool(pool).repayWithATokens(...)` but, unlike every other state-changing function in the contract, does not have the `nonReentrant` modifier. The Aave pool is an external contract that can invoke arbitrary logic (including reentering the connector or the `AccountingManager`) during the repayment. There is no `nonReentrant` on the Aave pool itself, and the connector's other functions rely on `nonReentrant` to prevent cross-function reentrancy.
- Impact: A reentrancy call into the connector or the AccountingManager during `repayWithATokens` can lead to inconsistent state between the Aave position, the registry, and the AccountingManager's internal accounting.

## SNXV3Connector: `claimRewards` lacks reentrancy guard
- Location: `contracts/connectors/SNXConnector.sol` : `claimRewards`
- Mechanism: `claimRewards` makes an external call to `SNXCoreProxy.claimRewards(...)` and then calls `_updateTokenInRegistry(collateralType)`. It is missing the `nonReentrant` modifier that is present on the other state-changing functions in the contract. The SNX core proxy is an external contract that can reenter the connector or the AccountingManager. A reentering call could trigger another `claimRewards` (or other functions) before the token-registry update has been recorded, potentially leading to double-claims or inconsistent position state.
- Impact: Reentrancy can be used to claim rewards multiple times or to manipulate the vault's view of the SNX position.

## Registry: EMERGENCY_ROLE can exercise maintainer and governer powers
- Location: `contracts/accountingManager/Registry.sol` : `onlyVaultMaintainer`, `onlyVaultMaintainerWithoutTimeLock`, `onlyVaultGoverner`
- Mechanism: Each of these modifiers is written as `if (msg.sender != vaults[_vaultId].<role> && hasRole(EMERGENCY_ROLE, msg.sender) == false) revert`. The logical operator is `&&`, meaning the modifier reverts only if the caller is neither the per-vault role nor the global EMERGENCY_ROLE. As a result, any address holding EMERGENCY_ROLE can call every function gated by these modifiers, including `addConnector`, `updateConnectorTrustedTokens`, `addTrustedPosition`, `removeTrustedPosition`, and `changeVaultAddresses`. The emergency role is typically expected to have a narrow, pause-oriented scope, not the ability to add connectors, change trusted tokens, or rotate governance addresses.
- Impact: Compromise or misuse of the EMERGENCY_ROLE allows the holder to add malicious connectors, whitelist arbitrary tokens for any connector (effectively approving them for use by the vault), change the vault's maintainer/governer to addresses they control, or remove trusted positions. This broadens the blast radius of an emergency-role compromise far beyond pausing the system.

## Registry: `addVault` enables AccountingManager as a connector before parameter validation of the AccountingManager itself
- Location: `contracts/accountingManager/Registry.sol` : `addVault`
- Mechanism: `addVault` sets `vault.connectors[vault.accountManager].enabled = true` and marks the base token as trusted on the AccountingManager connector before any of the require checks on the other addresses have been performed. The order of operations also stores the provided addresses in the vault struct first, so a later require failure leaves the vault struct partially populated (the accountManager, baseToken, governer, maintainer, etc. are all written before the final `vault.holdingPositions.push(...)` and the `emit`). Although the function reverts on require failure, this ordering means a future change to the require list could easily allow a partial-vault state. More importantly, the AccountingManager is enabled as a connector with the full base token pre-trusted, which means any subsequent re-initialization or misconfiguration of the AccountingManager could allow it to act as a fully-trusted token router for the base token.
- Impact: If the AccountingManager is later upgraded or replaced without removing this entry, the old AccountingManager (if still callable) retains full base-token trust. Combined with the EMERGENCY_ROLE bypass above, an attacker controlling the emergency role could swap in a malicious AccountingManager that the registry still trusts for the base token.

## AccountingManager: `recordProfitForFee` and `checkIfTVLHasDroped` interaction can forfeit accrued profit without resetting `totalProfitCalculated`
- Location: `contracts/accountingManager/AccountingManager.sol` : `recordProfitForFee`, `checkIfTVLHasDroped`, `collectPerformanceFees`
- Mechanism: When `checkIfTVLHasDroped` is invoked and the current profit is below `storedProfitForFee`, it sets `preformanceFeeSharesWaitingForDistribution = 0` and `profitStoredTime = 0`, but does **not** touch `totalProfitCalculated`. The next call to `recordProfitForFee` then computes `preformanceFeeSharesWaitingForDistribution = previewDeposit(((storedProfitForFee - totalProfitCalculated) * performanceFee) / FEE_PRECISION)` where `storedProfitForFee` is the freshly observed (and now lower) profit. Because `totalProfitCalculated` was not reduced, the difference `storedProfitForFee - totalProfitCalculated` can be negative in practice if TVL has dropped below the already-collected baseline, in which case the function's `if (storedProfitForFee < totalProfitCalculated) return;` guard fires and the manager gets no fees. However, if TVL recovers partially (but not to the previous `totalProfitCalculated`), the manager can again record profit, and the formula effectively double-counts the recovery: the manager receives fees on the recovery even though a portion of the underlying gain was forfeited earlier. More critically, an attacker (or a mis-timed keeper) can call `checkIfTVLHasDroped` immediately after `recordProfitForFee` but before the 12-hour lock expires, zeroing out the pending shares. While the 12-hour lock prevents `collectPerformanceFees` from running, the attacker can repeat this dance to ensure the manager never actually collects fees on volatile gains, while the share-inflation attack described below is still possible.
- Impact: The performance-fee accounting can be gamed: the manager can be denied legitimate fees, and conversely, if the manager times `recordProfitForFee` right after a temporary oracle spike (within the 12-hour window) and TVL does not drop, they collect fees on spike-driven TVL that subsequently reverts. The lock window only protects against the drop, not against the spike-then-harvest pattern.

## AccountingManager: First-mover advantage in deposit queue can grief other depositors
- Location: `contracts/accountingManager/AccountingManager.sol` : `calculateDepositShares`, `executeDeposit`
- Mechanism: `calculateDepositShares` iterates from `depositQueue.middle` forward as long as the record time of the entry is at or before the `oldestUpdateTime` of the vault's holding positions. The `oldestUpdateTime` is determined by the manager's recent updates to holding positions. A manager (or anyone able to call `updateHoldingPosition` on a connector, which is any active connector) can delay updating positions to keep `oldestUpdateTime` recent, so that `calculateDepositShares` makes no progress for arbitrary lengths of time. Because `executeDeposit` requires the calculation time to be at least `depositWaitingTime` (30 minutes) old, a depositor whose shares are never calculated can never have their deposit executed. A user who deposits just before a large `calculateDepositShares` + `executeDeposit` batch can also be skipped over if the manager processes only a subset, leaving the user's deposit stuck.
- Impact: Depositors can be indefinitely denied execution of their deposits, effectively locking their base tokens in the vault. The manager has unilateral control over when depositors receive shares.

## AccountingManager: `withdraw` does not revert on insufficient available liquidity
- Location: `contracts/accountingManager/AccountingManager.sol` : `withdraw`
- Mechanism: `withdraw` only checks the caller's share balance and adds the request to the queue. It does not check whether the vault has, or is likely to have, sufficient base-token liquidity to honor the withdrawal. Users can queue arbitrarily large withdrawals, and the only backstop is the `fulfillCurrentWithdrawGroup` check against `neededAssetsForWithdraw()`. A large number of withdrawals queued when liquidity is scarce can push the system into a state where `executeWithdraw` must pay users pro-rata from `totalABAmount` (which is capped at available assets), so users receive less than they expected. The `withdrawFee` is still taken on the full `baseTokenAmount` (calculated at the original higher amount), so the protocol collects fees on value it never actually disbursed.
- Impact: The protocol can collect withdrawal fees on withdrawals that are not fully honored, effectively taxing users for a failed (pro-rated) withdrawal.

## CompoundConnector: Trust check performed after external withdrawal
- Location: `contracts/connectors/CompoundConnector.sol` : `withdrawOrBorrow`
- Mechanism: The function calls `IComet(_market).withdraw(asset, amount)` before checking `if (!registry.isTokenTrusted(vaultId, asset, address(this)))`. While the `nonReentrant` modifier ensures the whole transaction reverts if the check fails, the external call to Comet happens first. If the Comet market is malicious or upgradeable in a way that lets it observe and react to the withdrawal before the check (e.g. by reentering via a token transfer), it could trigger state changes in the connector that are not rolled back by the require (e.g. events emitted, off-chain observers notified). More practically, the ordering wastes gas on a withdrawal that will revert and, in a future refactor that drops the `nonReentrant` or adds a try/catch, would leave the position exposed.
- Impact: A revert after an external withdrawal is a foot-gun for future modifications; the check should be hoisted before the external call to fail fast and reduce the attack surface.

## MorphoBlueConnector: `borrow` health-factor check uses post-borrow state and is called twice
- Location: `contracts/connectors/MorphoBlueConnector.sol` : `borrow`
- Mechanism: `morphoBlue.borrow(...)` is called first, and only afterwards is the health factor checked with `getHealthFactor`. If the check fails, the transaction reverts and the borrow is rolled back, so this is not directly exploitable, but the function reads `morphoBlue.market(id)` twice in the same line, which means it makes two external calls for the same data. Additionally, the health-factor check is performed against the post-borrow state, which is correct, but any oracle or market-state manipulation that happens between the borrow and the check could theoretically be exploited if the market's view of the position changes within the same transaction (e.g. via a callback). The function lacks an explicit reentrancy guard distinct from the implicit one in Morpho, relying solely on `nonReentrant` here.
- Impact: Minor gas waste and a fragile ordering; not directly exploitable in the current code, but a likely source of bugs in future modifications.

## BalancerConnector: TVL calculation uses only one token and assumes balanced weights
- Location: `contracts/connectors/BalancerConnector.sol` : `_getPositionTVL`
- Mechanism: The function reads only `pool.tokens[pool.tokenIndex]` and its weight, computes the value of that single token's share of the pool, and scales it by `lpBalance / _weight / _totalSupply`. This implicitly assumes that all other tokens in the pool maintain their weights relative to the indexed token. In an imbalanced Balancer pool (e.g., after a large swap or a token depeg), the other tokens may be worth significantly more or less than their weight-implied share, so the reported TVL diverges from the true redeemable value. The oracle price for the indexed token is also a single point of failure.
- Impact: TVL is manipulable by unbalancing the pool, which affects `getProfit()` and therefore performance-fee accounting, as well as the `previewDeposit` / `previewRedeem` calculations used for share pricing.

## Registry: `updateHoldingPosition` swap-and-pop can corrupt the `holdingPositions` array if the removed entry is the dummy
- Location: `contracts/accountingManager/Registry.sol` : `updateHoldingPosition` (public)
- Mechanism: The public `updateHoldingPosition` function computes `positionIndex = vault.isPositionUsed[holdingPositionId]` and, if the position exists, swaps the last element into the freed slot. The dummy entry at index 0 is pushed in `addVault` and is never registered in `isPositionUsed`. The code assumes that any non-zero `positionIndex` refers to a real position. However, because the dummy's `data` field is `""` and its `positionId` is `bytes32(0)`, a connector that accidentally calls `updateHoldingPosition` with `positionId = bytes32(0)` and `_data = ""` would compute `holdingPositionId = keccak256(abi.encode(msg.sender, bytes32(0), ""))`, which could collide with a previously registered position if any existed. More importantly, if a future code change allows a real position to be at index 0 (e.g. by removing the dummy), the swap-and-pop logic would corrupt the array because the check `positionIndex < vault.holdingPositions.length - 1` would try to move the last element into index 0 and then pop, leaving a gap or an inconsistent state if the array length was 1.
- Impact: A future refactor that removes the dummy position (or a code path that bypasses it) can corrupt the holdingPositions array, leading to incorrect TVL reporting and lost positions.

## BalancerFlashLoan: `caller` storage variable is not cleared on revert
- Location: `contracts/connectors/BalancerFlashLoan.sol` : `makeFlashLoan`, `receiveFlashLoan`
- Mechanism: `caller` is set to `msg.sender` in `makeFlashLoan` and cleared to `address(0)` after the flash loan completes. However, if the `vault.flashLoan(...)` call reverts inside `makeFlashLoan`, the `caller = address(0)` line is never reached, so `caller` remains set to the previous caller. Since `caller` is a storage variable, it persists across transactions. The `receiveFlashLoan` function, which is the only consumer of `caller`, is only called by the Balancer vault during a successful flash loan, so in normal operation this is fine. But if a custom or test Balancer vault is used (or if the function is called via a mocked vault), the stale `caller` could be used to bypass the `caller == keeperContract` check.
- Impact: A stale `caller` could be exploited in a test or upgrade scenario where `receiveFlashLoan` is invoked outside the normal Balancer vault flow, bypassing the keeper authorization check.

## NoyaFeeReceiver: `withdrawShares` creates a withdraw request owned by the receiver, but the receiver's shares are burned from the receiver during execution
- Location: `contracts/accountingManager/NoyaFeeReceiver.sol` : `withdrawShares`, `AccountingManager.executeWithdraw`
- Mechanism: `NoyaFeeReceiver.withdrawShares` calls `AccountingManager.withdraw(amount, receiver)`, which creates a `WithdrawRequest` with `msg.sender` (the NoyaFeeReceiver) as the owner. During `executeWithdraw`, `withdrawRequestsByAddress[data.owner]` is decremented and `data.owner` is burned. The `receiver` field in the withdraw request is the destination of the base-token payout, which is the NoyaFeeReceiver's configured `receiver` address. The accounting is internally consistent. However, the NoyaFeeReceiver's `burnShares` function calls `AccountingManager.burnShares`, which burns the NoyaFeeReceiver's own shares directly without going through the withdraw queue. This is by design (for fee realization), but it means the NoyaFeeReceiver can burn its shares at any time, including while a withdraw request is pending, potentially leaving the withdraw request pointing to a NoyaFeeReceiver with fewer shares than requested. The `withdrawRequestsByAddress` check in `_update` would block the NoyaFeeReceiver from transferring away the remaining shares, but it would not prevent the NoyaFeeReceiver from being left with an unfulfillable withdraw request.
- Impact: A NoyaFeeReceiver that burns its shares after initiating a withdraw request can leave the request unfulfillable, effectively trapping the corresponding base tokens in the protocol. The protocol's `executeWithdraw` would still attempt to burn the requested amount, but the NoyaFeeReceiver's balance might be insufficient, and the `withdrawRequestsByAddress` mapping might underflow (the check is `balanceOf(from) < amount + withdrawRequestsByAddress[from]`, but the burn in `executeWithdraw` does not re-check this).

## PendleConnector: `_getPositionTVL` uses a static router call that can revert on illiquid markets
- Location: `contracts/connectors/PendleConnector.sol` : `_getPositionTVL`, `getYTValue`
- Mechanism: The TVL calculation calls `staticRouter.swapExactYtForSyStatic(market, balance)`, which is a view function that simulates a swap. If the market is in a state where the simulation reverts (e.g., expired market, zero liquidity, or oracle failure), the entire TVL calculation reverts. Since `AccountingManager.TVL()` calls into the registry's `getPositionTVL`, which in turn calls `_getPositionTVL`, a single illiquid Pendle position can make the entire `TVL()` call revert, breaking `previewDeposit`, `previewRedeem`, `getProfit`, and all fee calculations.
- Impact: A single illiquid Pendle market can DoS the entire vault's accounting, preventing new deposits, blocking withdrawals, and freezing performance/management fee collection.

## LidoConnector: `claimWithdrawal` ETH balance calculation
- Location: `contracts/connectors/LidoConnector.sol` : `claimWithdrawal`
- Mechanism: The function records `beforeClaimBalance = address(this).balance` and then calculates the received ETH as `address(this).balance - beforeClaimBalance` after the claim. The connector has a `receive` function that accepts ETH, and the Lido withdrawal sends ETH to the caller. However, if any other ETH is sent to the contract between the balance read and the claim (e.g. via a selfdestruct or a direct transfer), the calculated amount is inflated, and the excess is deposited as WETH. This is a minor accounting issue, but in the context of a vault, it can lead to WETH being minted from thin air and credited to the vault, distorting TVL.
- Impact: ETH forced into the contract via `selfdestruct` or direct transfers during a withdrawal claim can be silently converted to WETH and counted as part of the position's value, inflating TVL and potentially enabling performance-fee extraction on value that did not belong to the vault.

## Summary
The most severe issues are in `BalancerFlashLoan` (broken return mechanism and unrestricted arbitrary external execution), `DolomiteConnector` (unsafe borrow without health checks), and `PendleConnector` (1-wei minimum output and DoS via static router). The Registry's broad EMERGENCY_ROLE bypass and the AccountingManager's manipulable performance-fee accounting are systemic risks. Several connectors lack reentrancy guards on functions that make external calls.
