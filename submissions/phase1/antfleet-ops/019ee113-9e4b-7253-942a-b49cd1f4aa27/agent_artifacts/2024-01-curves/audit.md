# Audit: 2024-01-curves

# Security Audit Report

## Broken access control modifiers in `Security.sol`
- Location: `Security.sol` : `onlyOwner`, `onlyManager`
- Mechanism: Both modifiers are written as bare comparison expressions with no revert. The statement `msg.sender == owner;` (and similarly `managers[msg.sender] == true;`) is a valid Solidity statement that evaluates to a `bool` and discards the result, then unconditionally runs `_;`. There is no `require`, `assert`, or `if (...) revert` to enforce the check, so the modifier is effectively a no‑op. Every function in `Curves` and `FeeSplitter` that uses these modifiers is therefore callable by any address, and `Security.transferOwnership` itself can be invoked by anyone to set a new owner (including `address(0)` to brick the system).
- Impact: Total compromise of every privileged function. An attacker can (1) call `setMaxFeePercent(1e18)` then `setProtocolFeePercent(1e18, attacker)` (or `setExternalFeePercent(1e18, 0, 0)`) to redirect 100% of every buy's fee component to themselves, (2) call `setFeeRedistributor` / `setERC20Factory` to a malicious contract, (3) call `transferOwnership(attacker)` to permanently take over admin rights, or (4) call `setManager` to whitelist themselves for `FeeSplitter.addFees` / `onBalanceChange`. The "subject‑only" functions (`onlyTokenSubject`) are still correctly enforced, so the attacker can additionally create a curve as the subject to collect subject/referral fee revenue after raising the fee parameters. Net effect is theft of a large fraction of all buy volume and complete admin takeover.

## `FeeSplitter.setCurves` is unprotected
- Location: `FeeSplitter.sol` : `setCurves`
- Mechanism: The function has no access control at all, so any external caller can overwrite the `curves` storage slot with a contract of their choosing. `balanceOf` and `totalSupply` in `FeeSplitter` blindly call `curves.curvesTokenBalance(...)` and `curves.curvesTokenSupply(...)` on whatever address is stored.
- Impact: An attacker can point `curves` at a malicious contract that returns fabricated values, e.g. returning `type(uint256).max` for `curvesTokenBalance(attacker, attacker)` so the attacker can drain the entire ETH balance via `claimFees`, or returning `0` for `curvesTokenSupply` so that `addFees` always reverts with `NoTokenHolders` and the splitter is bricked. Combined with the broken `onlyManager` above, the attacker can also call `addFees` themselves and self‑distribute.

## `FeeSplitter` user offsets are never updated on `transferCurvesToken` / `transferAllCurvesTokens` / `withdraw` / `deposit`
- Location: `Curves.sol` : `_transfer` (called from `transferCurvesToken`, `transferAllCurvesTokens`, `withdraw`, `sellExternalCurvesToken`'s inner `deposit` → `_transfer`, and `sellCurvesToken` indirectly via sell‑then‑deposit patterns)
- Mechanism: The `FeeSplitter` tracks each user's `userFeeOffset` so that on `claimFees` the user is credited `(cumulativeFeePerToken - userFeeOffset) * balance`. `onBalanceChange` is only called inside `_transferFees` (for the trader), and `updateFeeCredit` is only called inside `claimFees` / `batchClaiming`. When a user simply transfers tokens via `transferCurvesToken`, neither the sender's pending fees are flushed nor the recipient's offset is set to the current `cumulativeFeePerToken`. Concretely, if Alice (offset = 0) claims, then transfers to Bob, Bob's offset is still 0; any later `addFees` therefore credits Bob the full per‑token accumulation from the start, even though Alice already extracted that share.
- Impact: Insolvency / value theft. Example trace for a single subject: Alice holds 100 tokens, `cum = 0`. 1 ETH is added → `cum = 1e16`. Alice claims 1 ETH (her offset becomes `1e16`). Alice transfers all 100 tokens to Bob (Bob's offset is still 0). 1 ETH is added → `cum = 2e16`. Bob calls `claimFees`: `getClaimableFees` returns `(2e16 - 0) * 100e18 / 1e18 = 2 ETH`. The contract only holds 1 ETH so `payable(msg.sender).transfer(claimable)` reverts and Bob is unable to claim; the 1 ETH Alice already withdrew is permanently "double‑counted." A patient attacker can alternate between two EOAs (claim → transfer → other side claims → transfer back) to siphon every newly added `holderFee` while the prior side's claim has already been paid. Net result: the holder‑fee pool can be drained beyond what was deposited, and honest holders can be DoS'd from claiming their legitimate share.

## Protocol fee is not forwarded on sells
- Location: `Curves.sol` : `_transferFees` (the `isBuy == false` branch)
- Mechanism: In the first sub‑block, `firstDestination` is `msg.sender` (the seller) and the seller is paid `sellValue = price - protocolFee - subjectFee - referralFee - holderFee`. The subsequent sub‑blocks only forward `subjectFee` to the subject, `referralFee` to the referral destination, and `holderFee` to `feeRedistributor`. The `protocolFee` portion is silently retained in the contract (or implicitly transferred to the seller), even though on buys the same `protocolFee` is correctly sent to `feesEconomics.protocolFeeDestination`.
- Impact: Asymmetric protocol revenue. Every sell loses the protocol's share; cumulative ETH equal to `protocolFeePercent * price` per sell is either stuck in `Curves` (no withdrawal function exists for the owner) or quietly paid to the seller. Over time this either traps a growing ETH balance with no recovery path or systematically under‑pays the configured protocol destination on every sell, contrary to the buy behaviour and the explicit accounting in `getFees`.

## `FeeSplitter.onBalanceChange` is callable by anyone (compounding the modifier bug)
- Location: `FeeSplitter.sol` : `onBalanceChange`
- Mechanism: `onBalanceChange` is declared `onlyManager`, but because `onlyManager` in `Security.sol` is a no‑op, any address can invoke it. The function unconditionally overwrites `userFeeOffset[account] = cumulativeFeePerToken` (and pushes a token into `userTokens[account]` if the account has any balance).
- Impact: A griefing / theft primitive that is independent of the transfer‑offset bug. An attacker can monitor `addFees` and immediately call `onBalanceChange(victim, victim)` for any holder, zeroing out their pending `(cum - offset) * balance` claimable. Because `updateFeeCredit` is only called from `claimFees` / `batchClaiming`, the wiped offset is not restored until the victim claims, at which point their accrued fees (which were just added) are correctly credited — but the attacker can race each new `addFees` and continually reset the victim's offset, so the victim can never accumulate a meaningful claimable balance while the attacker keeps doing this. This is a direct denial of holder fees and is fully enabled by the broken `onlyManager` modifier.
