# Audit: 2023-07-pooltogether

# Security Audit Findings

## Inflated Available Yield via Direct Transfer (Donation Attack)
- **Location**: `vault/src/Vault.sol` : `_totalAssets` (~line 566), `availableYieldBalance` (~line 234), `liquidate` (~line 360)
- **Mechanism**: `_totalAssets()` is defined as `_yieldVault.maxWithdraw(address(this)) + super.totalAssets()`, i.e., the vault's own ERC20 balance is added to the yield vault's reported withdrawable amount. This balance is then used by `availableYieldBalance()` to compute how much yield can be liquidated (`_assets - _convertToAssets(_totalShares(), Down)`). Anyone can simply `transfer` the underlying asset directly to the Vault, which inflates `availableYieldBalance` without depositing anything into the YieldVault. In `liquidate()`, the code only forwards the vault's local balance into the YieldVault when `_amountOut >= _vaultAssets`. If the liquidator liquidates an amount smaller than the donation, the donation stays in the Vault, the liquidator's freshly-minted shares are entirely backed by the YieldVault, and the YieldVault can be drained below the principal owed to existing depositors. There is no check that the cap (`_liquidatableBalanceOf`) reflects only YieldVault-side yield.
- **Impact**: An attacker can grief or economically exploit the pool by donating the underlying token and then routing a liquidation through any LiquidationPair for an amount up to the inflated `availableYieldBalance`. The YieldVault is drained by `amountOut` (which can exceed the real accrued yield), causing the exchange rate to fall below 1 and existing depositors to lose principal proportionally. The attacker captures the difference between drained assets and the cost of the donation plus the LiquidationPair's spread.

## Reentrancy on Yield Vault / Prize Pool External Calls
- **Location**: `vault/src/Vault.sol` : `_deposit` (~line 501), `_withdraw` (~line 560), `liquidate` (~line 360)
- **Mechanism**: The vault performs multiple external calls (`_yieldVault.deposit`, `_yieldVault.withdraw`, `_prizePool.contributePrizeTokens`, `_prizePool.claimPrize`) and only afterward mints/burns shares or transfers assets. In `_deposit`, the user has already transferred assets into the vault before `_yieldVault.deposit` is called, but `_mint` has not yet run. In `_withdraw`, the user's shares are burned before `_yieldVault.withdraw` and `safeTransfer` execute. In `liquidate`, `_prizePool.contributePrizeTokens` and `_yieldVault.deposit` are both called before `_mint`. The contract has no reentrancy guard, so a malicious or hook-enabled yield vault or prize pool (or an ERC-777 underlying) can re-enter `deposit`, `withdraw`, `redeem`, or `liquidate` while the vault's accounting is mid-flight (assets present, shares not yet minted, or vice versa), allowing theft of assets or shares belonging to other users.
- **Impact**: If the YieldVault or PrizePool is upgradeable, hook-enabled, or otherwise untrusted, an attacker can use the reentrancy window to call `redeem`/`withdraw` on other users' shares, double-mint via `deposit`, or otherwise drain the vault. The effect on honest users is loss of principal or yield.

## Exchange Rate Returns Par When Yield Vault Is Empty
- **Location**: `vault/src/Vault.sol` : `_currentExchangeRate` (~line 698)
- **Mechanism**: The function falls back to returning `_assetUnit` (i.e., 1) in two distinct situations: when `totalSupply == 0` (legitimate "no shares yet" case) and when `_withdrawableAssets == 0` but `totalSupply > 0` (the yield vault is empty while shares are outstanding). The condition is `if (_totalSupplyAmount != 0 && _withdrawableAssets != 0)`, so the `else` branch returns `_assetUnit` even though shares are circulating against an empty YieldVault. `_isVaultCollateralized()` therefore reports `true`, and `maxDeposit` / `maxMint` / `liquidate` remain open.
- **Impact**: Users are told the vault is fully collateralized and may deposit or have liquidations occur when the YieldVault actually holds nothing. Subsequent `redeem`/`withdraw` calls will fail (or, if the YieldVault is a malicious implementation that lies about `maxWithdraw`, allow draining). The vault should report a sub-par rate (or revert) when the YieldVault cannot satisfy outstanding shares.

## `_currentExchangeRate` Capping Logic Subtly Wrong Under Loss
- **Location**: `vault/src/Vault.sol` : `_currentExchangeRate` (~line 698)
- **Mechanism**: When `_withdrawableAssets > _totalSupplyToAssets` the code does `_withdrawableAssets = _withdrawableAssets - (_withdrawableAssets - _totalSupplyToAssets)`, which algebraically equals `_totalSupplyToAssets`. This caps the rate at the last recorded rate, but it also means the rate can never rise above the last recorded rate even after the YieldVault recovers from a temporary loss. Because `_lastRecordedExchangeRate` is only updated on mint/burn, the cap "bakes in" the worst observed rate. Combined with the previous finding (par fallback when the vault is empty), the rate recovery path is also broken: an undercollateralized vault that recovers can only climb back to the last loss level, not to 1.
- **Impact**: Existing depositors do not benefit from a YieldVault that recovers after a loss; the improved value is silently captured by the next liquidator/mint. This is an accounting/MEV leak rather than a direct theft, but it materially mis-prices share redemptions after a recovery.
