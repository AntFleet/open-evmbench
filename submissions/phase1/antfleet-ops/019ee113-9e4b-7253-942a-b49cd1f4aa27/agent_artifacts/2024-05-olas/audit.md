# Audit: 2024-05-olas

# Security Audit Findings

## getPastVotes returns inflated voting power for blocks before user's first lock point
- Location: `governance/audits/internal/analysis/contracts/VotingEscrow-flatten.sol` : `VotingEscrow.getPastVotes` (called via `VotingEscrow._findPointByBlock`)
- Mechanism: `getPastVotes` calls `_findPointByBlock(blockNumber, account)` to locate the user checkpoint applicable to the queried block, then linearly extrapolates the bias: `uPoint.bias -= uPoint.slope * (blockTime - uPoint.ts)`. When a user has at least one checkpoint but the queried block is earlier than *all* of their checkpoints, the binary search in `_findPointByBlock` returns the first (earliest) checkpoint whose `blockNumber` is actually greater than the queried block. The subsequent extrapolation runs *backwards in time* (blockTime < uPoint.ts), causing `blockTime - uPoint.ts` to be negative, and the subtraction becomes an *addition*: `bias += slope * (uPoint.ts - blockTime)`. This yields a large positive (incorrect) voting power for a time at which the user held no lock at all. The companion function `balanceOfAt` performs the correct guard `if (uPoint.blockNumber < (blockNumber + 1))`, but `getPastVotes` omits this check.
- Impact: A user can vote on a Governor proposal with voting power they did not actually have at the snapshot block. Concrete exploit: wait for a proposal to be created (which fixes the snapshot block in the past), then create a ve-lock and immediately cast a vote. The OLA Governor reads voting power via `GovernorVotes._getVotes` -> `token.getPastVotes(snapshotBlock)`, so the inflated value is accepted as legitimate voting weight, allowing the attacker to swing proposals with tokens locked only after the snapshot was taken.

## _verifyBridgedData reads target address from wrong offset
- Location: `governance/audits/internal/analysis/contracts/GovernorOLA-flatten.sol` (the embedded `VerifyBridgedData._verifyBridgedData` in the flattened import chain) : `_verifyBridgedData`
- Mechanism: The data format is documented as `address target (20) | uint96 value (12) | uint32 payloadLength (4) | bytes payload`, but the inline assembly first advances the cursor by 20 and *then* calls `mload(add(data, i))` to read the target. As a result, the 20 bytes that are read as the "target" are actually the value field (12 bytes) + the leading bytes of the payloadLength/payload. The subsequent read of `payloadLength` at the new offset then reads from inside the payload. The intended layout is therefore misaligned by 20 bytes. (Note: the in-tree `ProcessBridgedDataArbitrum` does not call this helper â€” it decodes the Arbitrum calldata via `abi.decode` â€” so the bug is latent today, but any future bridge verifier inheriting `VerifyBridgedData` and using `_verifyBridgedData` will inherit this misalignment.)
- Impact: When invoked, `_verifyBridgedData` will verify authorization against a *garbage* target and a *garbage* selector extracted from a wrong offset. An attacker who can craft a bridged payload could either (a) bypass the allow-list entirely by ensuring the bytes read as "target" happen to be one of the authorized targets for whatever selector the assembly happens to land on, or (b) cause a DoS by producing a zero address that triggers `revert ZeroAddress()`. Because the function is invoked via `delegatecall` from `GuardCM._verifySchedule`, a successful bypass would let an arbitrary L1â†’L2 message be approved by the CM guard.

## OLA `mint` silently succeeds when inflation cap is exceeded
- Location: `governance/audits/internal/analysis/contracts/OLA-flatten.sol` : `OLA.mint`
- Mechanism: `mint` calls `inflationControl(amount)`, which returns a `bool`, and only invokes `_mint` when the check passes. If the check fails (i.e. the requested amount exceeds the inflation remainder), the function returns normally having done nothing â€” no event is emitted, no state change occurs, and the caller has no way to distinguish "minted 0" from "rejected". A legitimate minter integrating with this contract therefore has no on-chain signal that the cap was hit, and off-chain accounting that relies on the `Transfer` event or a return value will be silently wrong.
- Impact: The privileged `minter` (currently a contract such as `buOLA`, but changeable by `owner`) believes it has issued tokens and may credit the recipient, push downstream messages, or burn counter-assets in response. Those actions will be inconsistent with the actual on-chain supply. The economic effect is bounded by the cap, but a long-running integration can drift significantly from intended state.

## GuardCM `paused` is a uint8 instead of an explicit two-state value
- Location: `governance/audits/internal/analysis/fuzzing/.../GuardCM-flatten.sol` (and its original source) : `GuardCM.paused`
- Mechanism: `paused` is declared `uint8` and is set to `1` (active) or `2` (paused). `checkTransaction` only enforces the restrictions when `paused == 1`. Any other value (including values reachable only via a hypothetical future setter that writes a wrong literal) disables all guard checks, because the only branch tested is equality with `1`.
- Impact: Today no code path sets `paused` to a value other than `1` or `2`, so this is defensive only. However, because the type is `uint8` and the semantics are not encoded in the type, a careless future change (e.g. `paused = 0` during initialization, an off-by-one, or a stray `paused = 3`) would silently turn the guard into a no-op for every CM transaction. A `bool` or a two-member `enum` would make this impossible to misconfigure.

## `GuardCM.IGovernor.state` is declared non-view despite the implementation being view
- Location: `governance/audits/internal/analysis/fuzzing/.../GuardCM-flatten.sol` : local `IGovernor` interface inside `GuardCM`
- Mechanism: The locally-declared `IGovernor` interface used by `GuardCM.pause` declares `function state(uint256) external returns (ProposalState)` with no `view` modifier, while the canonical `GovernorOLA.state` is `public view`. The call still succeeds at runtime, but the type mismatch means static analyzers, ABI decoders, and off-chain consumers that trust the interface signature will treat the call as a state-mutating transaction, and the `IGovernor` re-declared here is not the one imported by the Governor stack.
- Impact: Low-impact specification drift. Tools that pre-compute calldata or gas estimates from the local interface will be wrong, and a future change that makes `Governor.state` actually consume gas (e.g. by reading from a non-view oracle) will silently break `GuardCM.pause` because the interface no longer matches. It also slightly widens the surface for accidental "interface vs implementation" bugs in cross-contract reviews.
