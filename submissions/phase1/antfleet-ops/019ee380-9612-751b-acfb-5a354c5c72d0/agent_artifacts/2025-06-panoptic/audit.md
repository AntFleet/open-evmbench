# Audit: 2025-06-panoptic

Below are the genuine security vulnerabilities I found. I focused on exploitable logic/accounting/access-control flaws.

## Zero‑fulfillment permanently bricks `executeDeposit` / `executeWithdrawal` (locked funds, burned shares)
- Location: `src/HypoVault.sol` : `executeDeposit` and `executeWithdrawal`
- Mechanism: Both functions divide by the epoch's *fulfilled* quantity using `Math.mulDiv`, which reverts when the denominator is `0`. In `executeDeposit`, `Math.mulDiv(userAssetsDeposited, _depositEpochState.sharesReceived, _depositEpochState.assetsFulfilled)` divides by `assetsFulfilled`. In `executeWithdrawal`, `Math.mulDiv(sharesToFulfill, _withdrawalEpochState.assetsReceived, _withdrawalEpochState.sharesFulfilled)` divides by `sharesFulfilled`. The contract's own design explicitly supports partial fulfillment and "moving the remainder to the next epoch," which means `fulfillDeposits(0, …)` / `fulfillWithdrawals(0, …, …)` are legitimate calls used to advance an epoch without disbursing. When an epoch is advanced with `0` fulfilled, `assetsFulfilled`/`sharesFulfilled` are stored as `0`, and every subsequent `execute*` call for that epoch reverts on the division before the carry‑forward branch is ever reached.
- Impact: Any deposit/withdrawal request that lands in a zero‑fulfilled epoch is permanently stuck. For withdrawals this is worse than a DoS: the shares were already burned from `balanceOf` in `requestWithdrawal` (and removed from `totalSupply` at fulfillment), but `executeWithdrawal` can never run, so the user permanently loses their shares with zero asset recovery. `cancelDeposit`/`cancelWithdrawal` only operate on the current (`depositEpoch`/`withdrawalEpoch`) epoch, so there is no recovery path for a past, zero‑fulfilled epoch.

## Manager can drain assets that are "reserved" for queued withdrawals (broken reservation invariant)
- Location: `src/HypoVault.sol` : `manage` (both overloads) vs. `reservedWithdrawalAssets`
- Mechanism: When `fulfillWithdrawals` runs it credits `reservedWithdrawalAssets += assetsReceived`, and `fulfillDeposits`/`fulfillWithdrawals` subtract this amount from NAV so the reserved assets are excluded from share pricing — the protocol treats these assets as set aside and guaranteed for withdrawers who still have to call `executeWithdrawal`. However, `manage` lets the manager make arbitrary external calls with arbitrary value/calldata from the vault, with no carve‑out preventing it from transferring out the underlying token (including the portion accounted as `reservedWithdrawalAssets`, and not‑yet‑fulfilled queued deposits). Nothing tracks or fences off the reserved balance against `manage`.
- Impact: A malicious or compromised manager (or one that simply over‑allocates into illiquid Panoptic positions) can move out the underlying token that backs already‑fulfilled withdrawals. When users later call `executeWithdrawal`, the `SafeTransferLib.safeTransfer` fails for lack of balance, so fulfilled withdrawals become unredeemable even though the user's shares are already gone. The "reserved" accounting gives users a false guarantee that is not enforced on‑chain.

## Insufficient virtual shares enable an inflation / first‑depositor share‑price attack
- Location: `src/HypoVault.sol` : constructor (`totalSupply = 1_000_000`) and `fulfillDeposits`
- Mechanism: The vault seeds only `1_000_000` virtual shares (with no backing assets; the `+1` in `totalAssets` is the only virtual asset) as inflation protection. Share issuance is `sharesReceived = Math.mulDiv(assetsToFulfill, totalSupply, totalAssets)`. Because NAV is computed from real balances (the final `IERC20Partial(underlyingToken).balanceOf(_vault)` term and pool balances), an attacker who becomes the dominant shareholder can directly transfer (donate) underlying tokens into the vault to inflate `totalAssets`. With only ~1e6 virtual shares, this is far too small to dominate real deposit sizes, so a later epoch's `assetsToFulfill * totalSupply / totalAssets` can round down to a tiny number (or zero) of shares for honest depositors.
- Impact: Honest depositors whose epoch is fulfilled while NAV is inflated receive disproportionately few (or zero) shares, while their assets are absorbed into NAV that backs the attacker's existing shares — a classic value‑transfer/inflation attack. The donated capital remains the attacker's claim (they hold the supply), so the attack is profitable when the victim deposit exceeds the rounding loss the attacker fronts.

## Manager‑supplied prices can skew NAV up to `maxPriceDeviation` on every valuation
- Location: `src/accountants/PanopticVaultAccountant.sol` : `computeNAV`
- Mechanism: The manager passes `ManagerPrices` (`poolPrice`, `token0Price`, `token1Price`) and the only validation is `Math.abs(managerPrice - twapFilter(...)) <= maxPriceDeviation` (`StaleOraclePrice`). The *accepted* manager price — not the TWAP — is then used as the actual valuation input: `managerPrices[i].poolPrice` drives `Math.getAmountsForLiquidity` for option leg valuation, and `token0Price`/`token1Price` drive the `convert0to1`/`convert1to0` conversions to underlying. Since the manager freely chooses any value inside the `±maxPriceDeviation` band and is the sole trigger of `fulfillDeposits`/`fulfillWithdrawals`, they can always pick the band edge that biases NAV in their favor for the action being taken.
- Impact: On each fulfillment the manager can bias the reported NAV by up to `maxPriceDeviation`, minting themselves cheaper shares during deposits or extracting more assets during withdrawals than the true TWAP‑based value supports, at the expense of other vault participants. The size of the theft per call is bounded by `maxPriceDeviation`, but it is repeatable and one‑directional.

## NAV position list can be double‑counted by listing a held position twice
- Location: `src/accountants/PanopticVaultAccountant.sol` : `computeNAV` (position‑list validation loop)
- Mechanism: The manager supplies `tokenIds[i]`. The integrity checks are: each listed position must have nonzero balance (`positionBalanceArray[j][1] == 0` → `IncorrectPositionList`) and the total leg count must equal `pools[i].pool.numberOfLegs(_vault)`. Neither check enforces uniqueness or that the set of listed positions equals the set actually held. A manager can list one held position twice (each instance passes the nonzero‑balance check and contributes its real leg count) while omitting a different held position of equal leg count, keeping `numLegs == numberOfLegs(_vault)`.
- Impact: The duplicated position's exposure is counted twice and the omitted position's exposure is dropped, letting the manager misreport NAV (in either direction depending on which positions are long/short). Combined with the manager's exclusive control of fulfillment, this is another lever to misprice shares/withdrawals.

---

Notes on things I checked and consider safe by design: the permissionless `executeDeposit`/`executeWithdrawal` and `requestDeposit`/`requestWithdrawal` are intentional; the cross‑epoch carry‑forward of deposit/withdrawal remainders reconciles in aggregate with the epoch‑level state set in `fulfillDeposits`/`fulfillWithdrawals`; `_transferBasis` moves cost basis proportionally so it can't be used to reset/evade the performance fee; and `requestWithdrawal`'s basis math reverts on `shares > balance` via underflow. The first two findings above are the highest severity.
