# Audit: 2024-01-renft

I audited the ReNFT-style rental protocol (Kernel/policy/module architecture with a Gnosis-Safe guard). The most serious issue is a calldata-offset error in the Guard that lets a renter strip the protocol's Stop module off their rental safe.

## Guard reads the wrong argument when validating `disableModule`, allowing removal of the Stop module
- Location: `smart-contracts/src/policies/Guard.sol` : `_checkTransaction` (the `gnosis_safe_disable_module_selector` branch), using `gnosis_safe_disable_module_offset` from `smart-contracts/src/libraries/RentalConstants.sol`
- Mechanism: For `disableModule(address prevModule, address module)`, the field the guard must validate is the *second* argument (`module`), which lives at calldata word index 2 → memory offset `0x44` (the same pattern used correctly for `transferFrom`'s `tokenId` at `0x64`, `approve`'s `tokenId` at `0x44`, etc.). However `gnosis_safe_disable_module_offset` is set to `0x24`, the offset of the *first* argument. `_loadValueFromCalldata(data, 0x24)` therefore returns `prevModule`, and `_revertNonWhitelistedExtension` checks whether `prevModule` (not the module being removed) is whitelisted. `enableModule(address module)` legitimately uses `0x24` because there its single argument is at index 0, which is why the constant looks plausible but is wrong for the two-argument disable selector.
- Impact: A rental-safe owner first enables any admin-whitelisted extension `E` (a normal, allowed action). The safe's module linked list is now `SENTINEL → E → stopPolicy`. The owner then calls `disableModule(prevModule = E, module = stopPolicy)`. The guard inspects `prevModule = E`, finds it whitelisted, and permits the call, so the Stop policy module is removed from the safe. `Stop.stopRent` reclaims rented assets via `ISafe(order.rentalWallet).execTransactionFromModule(...)`, which only works while Stop is an enabled module. Once removed, the lender can never reclaim their NFT — the rented asset is permanently frozen in the renter's safe, i.e. loss of the lender's collateralized asset. (The guard still blocks direct transfers and `setGuard`, so this manifests as permanent lock-up/denial of the lender's property rather than a clean steal-and-sell, but the lender irrecoverably loses the asset.)

## `RentalOrder` EIP-712 hash omits the `rentalWallet` field declared in its type string
- Location: `smart-contracts/src/packages/Signer.sol` : `_deriveRentalOrderHash` (vs. the `rentalOrderTypeString` in `_deriveRentalTypehashes`)
- Mechanism: The `RentalOrder` type string registered in `_RENTAL_ORDER_TYPEHASH` declares nine members including `address rentalWallet`, but the `abi.encode` in `_deriveRentalOrderHash` only encodes eight members and skips `order.rentalWallet`. The produced struct hash thus does not commit to the rental wallet at all, even though the typehash claims it does. This hash is the storage key for active rentals (`STORE.addRentals` in `Create` and `STORE.removeRentals`/`removeRentalsBatch` in `Stop`).
- Impact: The on-chain order-hash → order binding does not authenticate `rentalWallet`. The protocol is saved from a practical collision only because `seaportOrderHash` (which *is* included) is unique per order; the `rentalWallet` passed to `stopRent` is instead implicitly constrained by the `rentedAssets` underflow check. Functionally this is a broken EIP-712 commitment: the declared typed-data structure does not match what is hashed, so any external party recomputing the canonical hash from the published struct (per the advertised type string) derives a different value than the contract, and the field that designates which safe holds the assets is unauthenticated in the order hash. Severity is limited by the `seaportOrderHash` uniqueness, but it is a genuine signing/accounting correctness flaw in security-critical hashing code.

I did not find exploitable reentrancy in `Stop.stopRent` (a reentrant second settle underflows `PaymentEscrow.balanceOf` and reverts the whole transaction), and the CREATE2 sender-binding, role gating, and pro-rata math checked out.
