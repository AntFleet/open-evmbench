# Audit: 2025-04-forte

I analyzed the two libraries (`Float128` arithmetic/comparison core and the `Ln` natural-log layer built on top of it). Below are the genuine logic vulnerabilities I can substantiate with a concrete code path.

## `le` / `ge` return `false` for mathematically-equal operands with different M/L encodings
- Location: `src/Float128.sol` : `le(packedFloat,packedFloat)` and `ge(packedFloat,packedFloat)`
- Mechanism: Both functions short-circuit to `true` only on exact bit equality (`packedFloat.unwrap(a) == packedFloat.unwrap(b)`). When the raw words differ they fall into the normalized-comparison body, which up-scales any 38-digit (M) operand to the 72-digit (L) domain (`aMan := mul(aMan, BASE_TO_THE_DIGIT_DIFF); aExp := sub(aExp, shl(EXPONENT_BIT, DIGIT_DIFF_L_M))`) specifically so values can be compared across encodings. The same numeric value can legitimately exist in both encodings — e.g. `toPackedFloat(1,0)` yields the M encoding (mantissa `1e37`, exp `-37`) while `toPackedFloat(10**71,-71)` (or any `divL`/`mul` result) yields the L encoding (mantissa `1e71`, exp `-71`) of the value `1`. After the M→L scaling both normalize to the identical `aExp`/`aMan`, so the body executes only `retVal := lt(aMan, bMan)` (for the non-negative branch) or `retVal := gt(aMan, bMan)` (negative branch), each of which is `false` for equal mantissas. Neither branch ever assigns `true` for the equal case, so the equality that `le`/`ge` are supposed to include is dropped whenever the two encodings are not bit-identical.
- Impact: For any consumer that relies on these as value comparisons (the intended use, given the cross-encoding normalization), an `a` and `b` that are numerically equal but stored in different mantissa sizes will report `a <= b == false` and `a >= b == false`. Boundary checks such as `require(price.ge(floor))`, `require(amount.le(cap))`, collateral/threshold gates, or sorted-insertion invariants silently fail exactly at the equality boundary, letting an attacker push a value to the exact limit and have the `>=`/`<=` guard reject (DoS) or, depending on branch direction, bypass it. `lt`/`gt` are not affected (their equal-case result of `false` is correct).

## `ln` does not validate the sign/domain of its input
- Location: `src/Ln.sol` : `ln(packedFloat)` → `ln_helper(...)`
- Mechanism: `ln` extracts the mantissa with `and(input, MANTISSA_MASK)` and the exponent, but never inspects the mantissa sign bit (`MANTISSA_SIGN_MASK`, bit 240) and never rejects a zero mantissa. The natural logarithm is only defined for strictly positive reals, yet a negative `packedFloat` is processed as though it were its absolute value, and a zero input flows into `ln_helper` where the magnitude path divides by / operates on a zero mantissa producing an unconstrained result rather than a revert (contrast with `Float128.div`/`sqrt`, which explicitly revert on division-by-zero / negative roots). 
- Impact: Callers computing `ln` of a value that can be non-positive receive a plausible-looking but meaningless finite result (the log of `|x|` for negatives, garbage for `0`) instead of a revert. Any pricing/curve/rate model that derives quantities from `ln` of a user-influenceable signed quantity can be steered with a negative or zero argument to obtain an arbitrary/incorrect output that the rest of the system treats as a valid logarithm, corrupting downstream accounting.

I did not find reentrancy, access-control, or external-call issues (these are pure libraries). The two issues above are the substantive logic/accounting defects: the comparison-equality drop in `le`/`ge` is the most directly exploitable as a boundary-check bypass.
