# Audit: 2024-05-loop

I analyzed `PrelaunchPoints.sol`. The collateral interfaces are trivial. The interesting attack surface is the 0x-swap path used in `claim`. Findings below.

## Swap-data validation reads hardcoded calldata offsets instead of ABI pointers
- **Location:** `src/PrelaunchPoints.sol` : `_validateData` / `_decodeUniswapV3Data` (and `_decodeTransformERC20Data`)
- **Mechanism:** `_decodeUniswapV3Data` decodes the `_data` blob with hand-rolled assembly that assumes a *fixed* layout: it reads the `encodedPath` length at `p+96` and the path bytes starting at `p+128`, i.e. it assumes the dynamic `bytes encodedPath` argument is stored immediately after the 4 static head words. It never follows the ABI offset pointer that actually lives at calldata position 4. The 0x `exchangeProxy`, however, decodes the *same* `_data` using standard ABI decoding, which **does** follow that offset pointer. An attacker can therefore set the offset word to point at a benign, validator-pleasing path (so `_validateData` parses `inputToken == _token`, `outputToken == WETH`, `recipient == address(this)`), while placing a *different* real `encodedPath` elsewhere in the buffer that 0x will actually execute. The entire purpose of `_validateData` — constraining the externally-supplied swap calldata that is then handed verbatim to `_fillQuote` → `exchangeProxy.call(_swapCallData)` — is defeated. The `transformERC20` branch has the related weakness that the `Transformation[] transformations` array (the part that actually decides what the swap does) is never inspected at all; only the three static head words are checked.
- **Impact:** A claimer controls the swap that the contract performs on its own pooled funds: arbitrary Uniswap path / arbitrary 0x transformers, routed through attacker-controlled pools, with no validated minimum output. At minimum this lets a claimer self-route through a pool they own and, combined with the balance-sweep below, manipulate proceeds; more broadly it removes the only guard standing between user-supplied calldata and an arbitrary call into the 0x proxy, so any residual/standing token approval or future proxy behavior becomes exploitable rather than contained.

## No minimum-output (slippage) check on claim swaps
- **Location:** `src/PrelaunchPoints.sol` : `_claim` (non-ETH branch) / `_validateData` / `_fillQuote`
- **Mechanism:** `_validateData` only checks the *input* side (input token, output token identity, input amount). It never validates `minBuyAmount` (UniswapV3) or `minOutputTokenAmount` (TransformERC20), and `_fillQuote` accepts whatever ETH the swap returns: `claimedAmount = address(this).balance`. The swap of a user's locked LRT into ETH happens entirely on-chain with no enforced floor on the received amount.
- **Impact:** Every `claim` is sandwichable. An MEV searcher can move the pool, let the claim execute at a poor rate, and back-run for profit, with the loss borne by the claimer. Because the proceeds are immediately converted 1:1 into lpETH for the user, the user permanently realizes the sandwiched value.

## Token claim credits the entire contract ETH balance to a single claimer
- **Location:** `src/PrelaunchPoints.sol` : `_claim`, line `claimedAmount = address(this).balance;`
- **Mechanism:** After the swap, the deposited amount is computed as the contract's *total* ETH balance, not the delta actually produced by `_fillQuote` (`boughtETHAmount` is computed in `_fillQuote` but discarded). The code relies on the comment "there should not be any ETH in the contract," but `receive()` is open and accepts ETH from anyone, and partial-fill / leftover ETH can remain.
- **Impact:** Any ETH resident in the contract at claim time (donated via `receive()`, dust left from a prior swap, or any ETH not swept by `convertAllETH`) is swept by whichever token-holder claims next and minted to them as lpETH, rather than going to its rightful owner. A claimer can deliberately time/donate to capture stray balances.

Notes that are *not* exploitable and I'm explicitly excluding: `_percentage > 100` reverts on the checked subtraction `userStake - userClaim`; the ETH `withdraw`/`_claim` paths follow checks-effects-interactions (balance zeroed before the external `call`), so no reentrancy; `lockFor`/`lockETHFor` only let a caller gift their own funds to `_for`. The accounting `userStake.mulDiv(totalLpETH, totalSupply)` for ETH claims is sound because `totalLpETH`/`totalSupply` are frozen at conversion.
