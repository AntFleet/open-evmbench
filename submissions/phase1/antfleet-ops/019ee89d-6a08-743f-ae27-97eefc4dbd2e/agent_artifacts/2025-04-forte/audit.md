# Audit: 2025-04-forte

## `le` and `ge` return false for mathematically-equal operands of different mantissa sizes
- Location: src/Float128.sol : `le` and `ge`
- Mechanism: Both functions early-return for *bit-identical* operands (`if (packedFloat.unwrap(a) == packedFloat.unwrap(b)) return true;`) and then, in the equal-exponent branch, decide the result with a **strict** comparison of the mantissas: `le` does `retVal := lt(aMan, bMan)` (positives) / `retVal := gt(aMan, bMan)` (negatives), and `ge` does `retVal := lt(bMan, aMan)` / `retVal := lt(aMan, bMan)`. The author assumed that if the bits weren’t identical the mantissas can’t be equal — but the same value has two valid encodings (M = 38 digits, L = 72 digits). `div(x,x)` yields `1.0` as M (`10^37`, exp −37) while `divL(x,x)` yields `1.0` as L (`10^71`, exp −71); these have different packed bits, so the short-circuit is skipped. The comparison code then size-normalizes M→L (`aMan *= 10^34`, `aExp -= 34`), producing **equal** `aExp` and `aMan`, at which point the strict `lt`/`gt` returns false. So `a.le(b)` and `a.ge(b)` both return `false` even though `a == b`. (`eq` is likewise raw-bit and returns `false` here.) Note `lt`/`gt` are *not* affected because returning false for equal values is correct for them.
- Impact: Any invariant or bound check written as `require(a.le(threshold))` / `require(a.ge(floor))` will spuriously revert (DoS) — or, where the inverted result is trusted, silently pass — whenever the two operands are equal but originate from different-precision operations (e.g. mixing `div`/`divL`), corrupting protocol accounting or locking funds.

## `sqrt(0)` halts the entire call with `stop()` instead of returning zero
- Location: src/Float128.sol : `sqrt`
- Mechanism: The zero short-circuit is implemented in inline assembly as `if iszero(a) { stop() }`. `stop()` emits the EVM `STOP` opcode, which terminates the **whole message-call frame** (committing all prior state changes and returning empty data) rather than returning `0` from the inlined internal function as intended. There is no `leave`/return-from-function semantics here. Consequently, when `sqrt` is fed a zero packedFloat, execution does not continue in the caller: pre-`sqrt` state mutations are persisted, all post-`sqrt` logic is skipped, and the call “succeeds” with empty returndata (a caller decoding a `uint256` will revert on the truncated returndata, or the top-level tx will silently no-op).
- Impact: An attacker who can drive the `sqrt` argument to zero (e.g. a zero price, zero reserve, or a zeroed difference) forces a transaction to silently terminate after partial state changes, bypassing intended checks/settlement and enabling inconsistent-state or denial-of-service exploitation of any dependent protocol.

## `ln` performs no domain validation and silently accepts non-positive inputs
- Location: src/Ln.sol : `ln`
- Mechanism: `ln` extracts the mantissa with `mantissa := and(input, MANTISSA_MASK)`, discarding the `MANTISSA_SIGN_MASK` bit, and never checks for a negative or zero input. For a negative `input` it therefore computes `ln(|input|)` and returns a normal finite (positive-domain) result instead of reverting. For `input == 0` (mantissa 0, exponent −8192) it skips the `ln(1)==0` special case and runs `ln_helper`, which walks the Taylor/range-reduction path on a zeroed mantissa and returns a finite garbage value (≈ the log of the smallest representable magnitude) instead of −∞/revert.
- Impact: A consumer that feeds a derived or attacker-influenced signed value into `ln` receives a plausible-looking but mathematically wrong number for negative/zero arguments, producing silent accounting drift (mispriced rates, broken curve math) that an attacker can steer for profit.
