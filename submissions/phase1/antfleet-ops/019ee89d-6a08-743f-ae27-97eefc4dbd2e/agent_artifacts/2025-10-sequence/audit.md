# Audit: 2025-10-sequence

## Explicit session signatures are not bound to the wallet, enabling cross-wallet replay
- Location: src/extensions/sessions/SessionSig.sol : `hashCallWithReplayProtection`
- Mechanism: The per-call hash that each explicit session signer signs is `keccak256(abi.encodePacked(chainid, payload.space, payload.nonce, callIdx, Payload.hashCall(call)))` — it commits to chain id, nonce space, nonce, call index and the call contents, but **not** to the wallet address. In `SessionSig.recoverSignature` the explicit branch recovers the session signer purely from this hash (`callSignature.sessionSigner = ecrecover(callHash, v, r, s)`), and `SessionManager.recoverSapientSignature` never verifies any signature over an EIP-712 op-hash that embeds `verifyingContract = wallet`. The only wallet-specific binding is that the `SessionManager` sapient leaf and the session configuration must be part of the wallet's `imageHash`. Consequently, when a session signer's sapient leaf weight alone meets the threshold (the standard "autonomous session" setup), the entire `execute(payload, signature)` calldata is wallet-agnostic: an identical `encodedSignature` validates on any other wallet that shares the same session configuration `imageHash` and is at the same `{space, nonce}`. Note the implicit path is *not* affected because `generateImplicitRequestMagic` mixes in `wallet`, so this is specific to explicit sessions.
- Impact: A relayer who observes an explicit-session transaction for wallet A can re-submit the same calls against any sibling wallet B that was provisioned with the same session key/permissions (same config hash) and identical nonce state, executing those calls (e.g., token transfers up to the session's value/usage limits) on B without authorization from B's owner.

## Undefined `behaviorOnError == 3` makes a reverting call report success and suppresses fallback handling
- Location: src/modules/Calls.sol : `_execute` (mirrored in src/Guest.sol `_dispatchGuest`, src/Estimator.sol `_estimate`, src/Simulator.sol `simulate`)
- Mechanism: `Payload.fromPackedCalls` decodes `behaviorOnError = (flags & 0xC0) >> 6`, which yields `0..3`, but the execution loops only handle `BEHAVIOR_IGNORE_ERROR (0)`, `BEHAVIOR_REVERT_ON_ERROR (1)` and `BEHAVIOR_ABORT_ON_ERROR (2)`. When a call fails with `behaviorOnError == 3`, none of the `if (!success)` branches match, so control falls through to `emit CallSucceeded(_opHash, i)` and the loop continues with `errorFlag` left `false`. A failed sub-call is therefore recorded on-chain as succeeded, and—because `errorFlag` is not set—any immediately following `onlyFallback` call is skipped even though the preceding call actually reverted, defeating the intended fallback/compensation semantics.
- Impact: Off-chain consumers and any contract logic relying on `CallSucceeded`/`CallFailed` events or on `onlyFallback` recovery steps can be misled into treating a reverted operation as successful (and into not running its compensating fallback), corrupting multi-step batched-execution flows that encode error handling via these behaviors.
