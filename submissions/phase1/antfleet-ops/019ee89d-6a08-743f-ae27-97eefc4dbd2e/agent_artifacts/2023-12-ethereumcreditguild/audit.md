# Audit: 2023-12-ethereumcreditguild

## SurplusGuildMinter slashes every staker after any historical loss, locking their staked CREDIT
- Location: src/loan/SurplusGuildMinter.sol : `getRewards`
- Mechanism: The named return `userStake` is used in the slash test **before** it is loaded from storage. The function executes `lastGaugeLoss = GuildToken(guild).lastGaugeLoss(term); if (lastGaugeLoss > uint256(userStake.lastGaugeLoss)) { slashed = true; }` and only on the *next* line does `userStake = _stakes[user][term];`. At the comparison, `userStake` is still the zero-initialized memory struct, so `userStake.lastGaugeLoss == 0` and `slashed` becomes true whenever `lastGaugeLoss(term) != 0` â€” i.e. whenever the term has *ever* recorded a loss, ignoring the user's own stored checkpoint (which is the whole point: slashing should only trigger for losses occurring after the user staked). When `slashed` is true, `getRewards` overwrites `_stakes[user][term]` with an all-zero struct, and `unstake` does `if (slashed) return;` before ever calling `withdrawFromTermSurplusBuffer`, so the user's CREDIT is never returned.
- Impact: Once any loss is recorded in a term, every SurplusGuildMinter staker in that term (including those who staked afterward, and with no new loss) is permanently force-slashed â€” their stake record is wiped (callable by anyone via `getRewards(victim, term)`), unclaimed GUILD rewards are zeroed, and their principal CREDIT stays trapped in the ProfitManager term surplus buffer recoverable only by a privileged `GUILD_SURPLUS_BUFFER_WITHDRAW` holder.

## Offboarding never clears `canOffboard`, enabling quorum-less re-offboarding and a permanent PSM redemption freeze
- Location: src/governance/LendingTermOffboarding.sol : `offboard` (interaction with `cleanup`)
- Mechanism: `offboard()` removes the gauge and does `nOffboardingsInProgress++`, but it never resets `canOffboard[term]`; that flag is cleared only in `cleanup()`. After an offboard the term is deprecated yet `canOffboard[term]` stays `true`. If governance later re-onboards the term (`proposeOnboard` only requires `!isGauge(term)`, which holds while deprecated), the stale flag lets *anyone* call `offboard(term)` again with no new poll â€” `removeGauge` succeeds because the gauge is live again and `nOffboardingsInProgress` is incremented a second time. But `cleanup()` can run only once per term because it sets `canOffboard[term] = false` and itself requires `canOffboard[term]`, so the second increment can never be decremented, leaving the counter permanently â‰Ą 1.
- Impact: A single unprivileged caller can repeatedly offboard any re-onboarded lending term without community support (governance bypass / griefing) and can desync `nOffboardingsInProgress` so it never returns to 0, leaving `SimplePSM.redemptionsPaused` stuck `true` forever â€” a denial of service on all CREDITâ†’pegToken redemptions.
