# Audit: 2024-01-init-capital-invitational

## Missing access control on LSD oracle configuration
- Location: contracts/oracle/LsdApi3OracleReader.sol : setDataFeedIds / setQuoteTokens / setMaxStaleTimes
- Mechanism: In the sibling `Api3OracleReader`, `PythOracleReader` and `InitOracle`, every price-source configuration setter is gated by `onlyGovernor`. In `LsdApi3OracleReader` only `setApi3OracleReader` carries the modifier; `setDataFeedIds`, `setQuoteTokens` and `setMaxStaleTimes` are declared plain `external` with no role check. `getPrice_e36` computes `price = rate_e18(dataFeedId) * getPrice_e36(quoteToken) / 1e18` and gates on `block.timestamp - timestamp <= maxStaleTime`, all of which an attacker now fully controls. An attacker sets `maxStaleTime` to `type(uint96).max` (defeating the freshness check) and points `dataFeedId`/`quoteToken` at a feed/token pair that yields an arbitrary `price_e36` for any LSD token using this reader as a source in `InitOracle`.
- Impact: An unprivileged attacker can set the price of any LSD collateral/debt token to an arbitrary value, then over-value collateral to borrow far beyond its worth (or mis-price debt to force liquidations), draining the lending pools.

## Unauthorized modification of other users' orders
- Location: contracts/hook/MarginTradingHook.sol : updateOrder
- Mechanism: `cancelOrder` correctly enforces `_require(order.initPosId == initPosId, ...)` after loading `initPosId = initPosIds[msg.sender][_posId]`, binding the order to the caller's position. `updateOrder` omits this check entirely: it only verifies the order is `Active`, that the caller owns *some* position (`initPosId != 0`), and that `_collAmt <= getCollAmt(callerInitPos, callerCollPool)` — a constraint evaluated against the *attacker's own* position, not the order's. It then writes `triggerPrice_e36`, `limitPrice_e36`, `collAmt` and `tokenOut` directly into `__orders[_orderId]` for an arbitrary `_orderId`. `tokenOut` is also written without the `== baseAsset || quoteAsset` validation that `_createOrder` enforces. An attacker creates one funded margin position, then calls `updateOrder(attackerPos, victimOrderId, …)` to set a victim's `limitPrice_e36` to an attacker-favorable value, `triggerPrice_e36` to the current mark price (so the trigger passes), and `collAmt` to the victim's full collateral.
- Impact: An attacker can rewrite any active stop-loss/take-profit order's price and size, then `fillOrder` it to seize the victim's collateral while repaying only the manipulated (minimal) amount, stealing the victim's position equity.

## USDY price returned per-token instead of per-wei (1e18 over-valuation)
- Location: contracts/oracle/usdy/UsdyOracleReader.sol : getPrice_e36
- Mechanism: The protocol's price convention is "USD_e36 per wei (smallest unit)": `Api3OracleReader`, `PythOracleReader` and `LsdApi3OracleReader` all normalize by `10**decimals`, and `InitCore.getCollateralCreditCurrent_e36` multiplies `toAmtCurrent(shares)` (in wei) by `price_e36`. `UsdyOracleReader.getPrice_e36` returns `IRWADynamicOracle.getPrice() * 1e18`, where `getPrice()` is the per-whole-token USD price scaled by 1e18 (per the contract's own comment). For 18-decimal USDY the correct per-wei value is `getPrice()` itself (i.e. `getPrice() * 1e18 / 10**18`), but the code never divides by `10**decimals`, yielding a result 1e18× too large.
- Impact: Any USDY collateral is valued 1e18× its real worth, letting a depositor of a trivial amount of USDY borrow and drain entire lending pools.

## Swapped repayShares / repayAmt in order fill accounting
- Location: contracts/hook/MarginTradingHook.sol : _calculateRepaySize / _calculateFillOrderInfo / fillOrder
- Mechanism: `_calculateRepaySize` is declared `returns (uint repayAmt, uint repayShares)` and assigns `repayShares` = share count and `repayAmt` = `debtShareToAmtCurrent(repayShares)` (token amount), so its return tuple is `(amount, shares)`. The caller `_calculateFillOrderInfo` destructures it as `(repayShares, repayAmt) = _calculateRepaySize(...)`, so its local `repayShares` receives the *amount* and `repayAmt` receives the *share count* — the two are transposed (the same transposition exists in `MarginTradingLens._fillOrder`). `fillOrder` then does `safeTransferFrom(msg.sender, address(this), repayAmt)` (pulling a *share count* worth of borrow token) and `IInitCore(CORE).repay(borrPool, repayShares, …)` (passing a *token amount* as the shares argument), and `amtOut` is computed from the share count instead of the borrow amount. Once interest accrues and amount-per-share diverges from 1, the tokens pulled into the hook are insufficient for `_repay`'s `safeTransferFrom(hook, pool, debtShareToAmtCurrent(sharesToRepay))`.
- Impact: Stop-loss/take-profit fills revert once any interest accrues (making the orders permanently unfillable so users' risk-management protection silently fails), and in the residual 1:1 window the position's debt is repaid in the wrong unit, corrupting debt-share accounting.
