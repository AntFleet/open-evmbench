# Audit: 2024-05-arbitrum-foundation

## Force inclusion omits the time-based (`delaySeconds`) sequencer delay window
- Location: src/bridge/SequencerInbox.sol : `forceInclusion`
- Mechanism: A delayed message is identified by both an L1 block number (`l1BlockAndTime[0]`) and an L1 timestamp (`l1BlockAndTime[1]`), and the sequencer's exclusive inclusion window is defined by *two* independent parameters, `delayBlocks` and `delaySeconds` (both set arbitrarily by the owner in `_setMaxTimeVariation`, with no enforced relationship between them). `forceInclusion` only enforces the block dimension — `if (l1BlockAndTime[0] + delayBlocks_ >= block.number) revert ForceIncludeBlockTooSoon();` — and never checks the timestamp dimension. The companion guard that should follow it (`if (l1BlockAndTime[1] + delaySeconds >= block.timestamp) revert ForceIncludeTimeTooSoon();`) is absent; tellingly, the `ForceIncludeTimeTooSoon` error is still imported at the top of the file but is dead/unused everywhere in the codebase, indicating the check was dropped. Because `delaySeconds` is routinely configured larger than `delayBlocks * 12s` to provide a wall-clock safety margin (the `DelayBuffer` design notes the operator wants to "wait threshold > finality time … to avoid L1 reorgs"), the seconds window is frequently the binding (later) constraint, so a message can satisfy the block check while the intended time window has not yet elapsed.
- Impact: Anyone can force-include a delayed L1→L2 message into the rollup inbox before the full time-based sequencer window has passed, cutting short the sequencer's guaranteed exclusivity window and eroding the wall-clock safety margin the protocol relies on against censorship/reordering and L1 reorgs.
