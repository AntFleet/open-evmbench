# Audit: 2024-03-coinbase

## Cross-chain replay of index-based owner removal can remove unintended owners and lock the account
- Location: `src/SmartWallet/CoinbaseSmartWallet.sol` : `validateUserOp` / `executeWithoutChainIdValidation` (combined with `MultiOwnable.removeOwnerAtIndex`)
- Mechanism: When `userOp.callData` targets `executeWithoutChainIdValidation` (selector `0xbf6ba1fc`), `validateUserOp` recomputes the hash via `getUserOpHashWithoutChainId(userOp)`, which excludes `block.chainid`. The resulting signature is therefore valid for the *same* `userOp` on every chain where the wallet shares this address. `canSkipChainIdValidation` permits `removeOwnerAtIndex` to be carried this way, but that function removes the owner sitting at a numeric `index`, not a specific owner identity. Owner sets diverge across chains whenever an `addOwnerAddress`/`removeOwnerAtIndex` is performed via a normal (non-replayable) op, or when a wallet is deployed on a new chain after the original chain’s owner set has already evolved. Once divergent, replaying a signed `removeOwnerAtIndex(i)` (observed from the mempool/another chain, with the matching `REPLAYABLE_NONCE_KEY` sequence) deletes whatever owner occupies index `i` on the target chain.
- Impact: An attacker (or any relayer) can replay a legitimately-signed owner-removal onto a chain where index `i` maps to a different—or the last—owner, removing an unintended owner and potentially leaving the account with no valid signer, permanently freezing its funds on that chain.

## `removeOwnerAtIndex` can delete the final owner, permanently bricking the account
- Location: `src/SmartWallet/MultiOwnable.sol` : `removeOwnerAtIndex`
- Mechanism: The function deletes `isOwner[owner]` and `ownerAtIndex[index]` with no check that at least one owner remains afterward. Removing the only remaining owner leaves `isOwner` empty for every key; thereafter `_checkOwner()` reverts for all callers, `_validateSignature` reverts/fails for every `ownerIndex` (empty `ownerAtIndex` → `InvalidOwnerBytesLength`), and no `UserOperation` can ever validate again. There is no recovery path (no admin, no social recovery).
- Impact: A single `removeOwnerAtIndex` call against the last owner (reachable directly, or unintentionally via the cross-chain replay above) irreversibly bricks the wallet, freezing all assets it holds.

## Paymaster has no aggregate liability accounting, making the “impossible” `postOp` assert reachable (insolvency / griefing)
- Location: `src/MagicSpend/MagicSpend.sol` : `validatePaymasterUserOp` / `postOp`
- Mechanism: `validatePaymasterUserOp` credits `_withdrawableETH[sender] += withdrawAmount - maxCost` while only guarding with the per-call check `address(this).balance < withdrawAmount`. ERC-4337 runs all validations in a bundle before any execution/`postOp`, so the balance is unchanged across validations and several distinct, validly-signed requests—each individually ≤ balance but summing to more than the balance—all pass. During execution/`postOp` the cumulative payouts (`withdrawGasExcess` plus `forceSafeTransferETH` in `postOp`) then exceed the contract balance and revert; a reverting first `postOp` causes the EntryPoint to re-invoke `postOp` with `PostOpMode.postOpReverted`, hitting `assert(mode != PostOpMode.postOpReverted)`. The code comment explicitly assumes this state is unreachable because solvency is “checked at the validation step,” but that check is per-request, not aggregate, so the documented invariant is false.
- Impact: A set of legitimately-signed withdraw requests whose total exceeds the live ETH balance can be bundled so that later payouts revert, leaving validated requests unpayable and triggering the assert—paymaster insolvency that strands user funds and griefs bundlers.
