# Audit: 2024-03-neobase

## De-whitelisting a market permanently freezes all funds deposited in its LiquidityGauge
- Location: `src/LendingLedger.sol` : `update_market` (the `require(lendingMarketWhitelist[_market], "Market not whitelisted")` guard), reached via `src/LiquidityGauge.sol` : `withdrawUnderlying` → `_afterTokenTransfer` → `sync_ledger`
- Mechanism: `LiquidityGauge.withdrawUnderlying` calls `_burn`, which invokes `_afterTokenTransfer`, which unconditionally calls `LendingLedger.sync_ledger(from, -amount)`. `sync_ledger` resolves the gauge to its underlying market via the reverse mapping and then calls `update_market(market)`, whose first line reverts if `lendingMarketWhitelist[market]` is false. When governance calls `whiteListLendingMarket(market, false, _)` it sets `lendingMarketWhitelist[market] = false` but never disables the gauge. From that point every gauge-token burn (and every transfer, since `_afterTokenTransfer` fires on both `from` and `to`) reverts inside `update_market`, so `withdrawUnderlying` can no longer execute. The underlying tokens already escrowed in the `LiquidityGauge` (`safeTransferFrom` on deposit) become unrecoverable by users.
- Impact: A single governance de-whitelist toggle bricks `withdrawUnderlying`/transfers for the affected gauge, permanently trapping every depositor's underlying tokens unless and until governance re-whitelists the exact same market.

## update_market values every caught-up epoch at the current gauge weight instead of that epoch's weight
- Location: `src/LendingLedger.sol` : `update_market`
- Mechanism: The accrual loop iterates per epoch (`epoch = (i / BLOCK_EPOCH) * BLOCK_EPOCH`) and reads the gauge weight via `gaugeController.gauge_relative_weight_write(_market, epochTime)`. However `epochTime` is computed as `referenceBlockTime + ((block.number - referenceBlockNumber) * averageBlockTime) / 1000`, i.e. from the *current* `block.number`, and is recomputed identically on every loop iteration rather than from the loop cursor `i` (the block whose epoch is being processed). Consequently, whenever the market is caught up across more than one epoch, all of those past epochs are credited with the gauge's *current-week* relative weight (`points_total[floor(now/WEEK)]`) rather than the relative weight that was actually in effect during each historical epoch, while `cantoPerBlock[epoch]` does vary correctly per epoch.
- Impact: CANTO reward distribution is mis-weighted across markets — a market whose relative weight changed between epochs has its historical rewards computed with the wrong (current) weight, letting a market that recently gained weight retroactively over-collect (or, conversely, under-collect) CANTO relative to what governance allocated for those past epochs.
