# Audit: 2025-06-panoptic

## Zero-fulfilled deposit epoch permanently bricks `executeDeposit` (division by zero) and strands depositor funds
- Location: `src/HypoVault.sol` : `executeDeposit` (interacting with `fulfillDeposits`)
- Mechanism: `executeDeposit` computes `userAssetsDeposited = mulDiv(queuedDepositAmount, assetsFulfilled, assetsDeposited)` and then `sharesReceived = Math.mulDiv(userAssetsDeposited, _depositEpochState.sharesReceived, _depositEpochState.assetsFulfilled)`. The second `mulDiv` uses `assetsFulfilled` as the denominator. `fulfillDeposits` accepts `assetsToFulfill == 0` on an epoch that still has `assetsDeposited > 0` (no underflow: `assetsRemaining = assetsDeposited - 0`), advances `depositEpoch`, and freezes that epoch with `assetsFulfilled = 0`. No function ever rewrites a past epoch's state, so `assetsFulfilled` for that epoch is `0` forever. Any depositor with `queuedDeposit[user][epoch] > 0` who calls `executeDeposit(user, epoch)` passes the `epoch >= depositEpoch` guard and then reverts on `mulDiv(..., 0)`. Their per-user balance can never be claimed and never rolls forward (only the reverting `executeDeposit` writes `queuedDeposit[user][epoch+1]`), even though `fulfillDeposits` already rolled the *global* `assetsRemaining` into the next epoch's `assetsDeposited` and will later mint the corresponding `sharesReceived` into `totalSupply`.
- Impact: A single `fulfillDeposits(0, …)` call against a non-empty epoch permanently locks every queued depositor's assets in that epoch and mints their share slice into `totalSupply` with no owner, silently transferring the trapped depositors' value to the remaining shareholders.

## Zero-fulfilled withdrawal epoch permanently bricks `executeWithdrawal` (division by zero) and strands withdrawer shares
- Location: `src/HypoVault.sol` : `executeWithdrawal` (interacting with `fulfillWithdrawals`)
- Mechanism: `executeWithdrawal` computes `assetsToWithdraw = Math.mulDiv(sharesToFulfill, _withdrawalEpochState.assetsReceived, _withdrawalEpochState.sharesFulfilled)`, using `sharesFulfilled` as the denominator. `fulfillWithdrawals` accepts `sharesToFulfill == 0` on an epoch with `sharesWithdrawn > 0` (`assetsReceived = mulDiv(0, …) = 0 <= maxAssetsReceived`, `sharesRemaining = sharesWithdrawn - 0`), advances `withdrawalEpoch`, and freezes that epoch with `sharesFulfilled = 0`. As with deposits, past-epoch state is never rewritten, so any withdrawer with `queuedWithdrawal[user][epoch].amount > 0` who calls `executeWithdrawal(user, epoch)` clears the `epoch >= withdrawalEpoch` guard and then reverts on the `mulDiv(..., 0)`. The per-user pending withdrawal can never be paid out and never rolls forward (only the reverting `executeWithdrawal` writes `queuedWithdrawal[user][epoch+1]`), while `fulfillWithdrawals` already removed those shares from `balanceOf` via the earlier `_burnVirtual` in `requestWithdrawal` and rolled the global `sharesRemaining` into the next epoch.
- Impact: A single `fulfillWithdrawals(0, …)` call against a non-empty epoch permanently locks every queued withdrawer's burned shares and the assets owed to them, with no recovery path, redistributing the stranded value to the remaining vault participants.
