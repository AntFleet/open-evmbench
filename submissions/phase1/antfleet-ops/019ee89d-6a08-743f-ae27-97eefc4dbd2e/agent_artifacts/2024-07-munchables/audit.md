# Audit: 2024-07-munchables

## Pet rewards inflated by an extra 1e18 scaling factor
- Location: src/managers/SnuggeryManager.sol : `pet`
- Mechanism: `PET_TOTAL_SCHNIBBLES` is a 1e18-denominated quantity â€” the inline config note reads `// 10e18 / 72`, and the bonus line `bonusSchnibbles = (PET_TOTAL_SCHNIBBLES * bonusPercent) / 1e18` only makes sense if `PET_TOTAL_SCHNIBBLES` is already 1e18-scaled (it divides the product of two 1e18-scaled values by 1e18). The reward then computes `petterSchnibbles = ((totalSchnibbles * 5) / 11) * 1e18` and `pettedSchnibbles = ((totalSchnibbles * 6) / 11) * 1e18`, multiplying an already-1e18-scaled number by a second `1e18`. Every other schnibble figure in the protocol (harvest in `AccountManager._harvest`, sprays, `LandManager._farmPlots`) stays at 1e18 scale, so the credited pet amounts are ~1e18Ă— too large. `pet` only requires both accounts to be registered and `_pettedMainAccount == _pettedOwner`, with cooldowns of 5 min/munchable and 10 min/petter.
- Impact: Any registered user, using two controlled accounts, can mint astronomically inflated `unfedSchnibbles` every 10 minutes, which feed into chonks â†’ claimable points â†’ MUNCH tokens via `ClaimManager.convertPointsToTokens`, enabling near-unlimited token/reward minting.

## transferToUnoccupiedPlot never updates the toiler's plotId (or clears dirty)
- Location: src/managers/LandManager.sol : `transferToUnoccupiedPlot`
- Mechanism: The function frees `plotOccupied[landlord][oldPlotId]` and sets `plotOccupied[landlord][plotId] = {occupied:true, tokenId}`, and updates `toilerState[tokenId].latestTaxRate`, but it never assigns `toilerState[tokenId].plotId = plotId` (and never resets `dirty` or `lastToilDate`). After the move, the toiler still records `plotId == oldPlotId`. Consequently `unstakeMunchable` executes `plotOccupied[_toiler.landlord][_toiler.plotId] = {false,0}` against the stale `oldPlotId`, leaving the *new* plot permanently flagged occupied, while the old plot is wrongly cleared. If another renter stakes into the freed `oldPlotId` after the transfer, the first renter's later unstake evicts that renter's occupancy record (whose toiler still points at `oldPlotId`), allowing multiple munchables to map to the same plot. Because `dirty` is also never reset here, a munchable frozen by the `_farmPlots` out-of-range branch cannot be revived through this (documented) recovery path.
- Impact: An attacker (or normal use) corrupts plot occupancy accounting â€” permanently bricking specific plots (`OccupiedPlotError` forever) and forcing double-occupancy/double-earning collisions across renters.

## Off-by-one in farm plot-range check lets munchables earn on non-existent plots
- Location: src/managers/LandManager.sol : `_farmPlots`
- Mechanism: Valid plot indices for a landlord are `0 .. _getNumPlots(landlord)-1`, so a plot is out of range when `plotId >= _getNumPlots(landlord)`. The freeze/dirty logic instead tests `if (_getNumPlots(landlord) < _toiler.plotId)`, using strict `<` rather than `<=`. When a landlord reduces their locked value so that `_getNumPlots(landlord) == _toiler.plotId`, the munchable's plot no longer exists, yet the condition is false, so the code keeps `timestamp = block.timestamp` and accrues `schnibblesTotal = (block.timestamp - lastToilDate) * BASE_SCHNIBBLE_RATE` instead of freezing accrual at `plotMetadata[landlord].lastUpdated`.
- Impact: A renter staked on the boundary plot keeps minting schnibbles after the landlord has shrunk plots below that index, over-issuing rewards that should have been frozen.

## Harvest applies the current USD price retroactively over the whole elapsed period
- Location: src/managers/AccountManager.sol : `_harvest` / `getDailySchnibbles`
- Mechanism: `_harvest` computes `dailySchnibbles` from `lockManager.getLockedWeightedValue(_caller)`, which reads the live `configuredTokens[...].usdPrice`, then multiplies by the full interval `block.timestamp - lastHarvestDate`. A price change executed by `LockManager._execUSDPriceUpdate` mutates `usdPrice` (and therefore `getLockedWeightedValue`) for all users without settling their pending harvest, so the new price is applied to the entire unharvested time window rather than only to time after the update. The USD-price proposal/approval flow is fully on-chain and observable before it executes.
- Impact: A locker can withhold `harvest()` until an approved price increase executes and then harvest, receiving the higher rate over the whole prior interval, inflating their accrued schnibbles (and downstream points/MUNCH) beyond what was actually earned.
