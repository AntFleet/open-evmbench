# Audit: 2024-01-renft

## Guard `disableModule` check reads the wrong calldata word, allowing removal of the Stop module
- Location: smart-contracts/src/libraries/RentalConstants.sol : `gnosis_safe_disable_module_offset` (consumed in smart-contracts/src/policies/Guard.sol : `_checkTransaction`)
- Mechanism: For `disableModule(address prevModule, address module)` the module actually being removed is the *second* argument, sitting at calldata byte offset 0x24, i.e. memory offset `0x20 + 0x24 = 0x44` inside the `data` bytes. The guard instead uses `gnosis_safe_disable_module_offset = 0x24`, so `_loadValueFromCalldata(data, 0x24)` loads the *first* argument (`prevModule`) and passes it to `_revertNonWhitelistedExtension`. The guard therefore only checks that the attacker-chosen `prevModule` is whitelisted, never the `module` being disabled. A rental-safe owner exploits this by first calling `enableModule(whitelistedExtension)` (permitted, because `gnosis_safe_enable_module_offset = 0x24` correctly reads the single argument and the extension is admin-whitelisted); Gnosis Safe inserts the new module at the front of the linked list, making `whitelistedExtension` the predecessor of the Stop policy. The owner then calls `disableModule(whitelistedExtension, stopPolicy)`: the guard reads `prevModule = whitelistedExtension`, finds it whitelisted, and allows the call, removing the Stop policy from the safe's modules. The correct constant is `0x44`, which would have made the guard validate `stopPolicy` and revert (Stop is not in `whitelistedExtensions`).
- Impact: Once the Stop module is removed, `Stop._reclaimRentedItems`' `execTransactionFromModule` call reverts permanently, so the rental can never be stopped; the lender's rented ERC721/ERC1155 (and the escrowed rent, which is only released on stop) is locked forever in the renter's safe while the renter keeps using the asset, and it cannot be recovered because re-enabling Stop is itself blocked (Stop is not a whitelisted extension).
