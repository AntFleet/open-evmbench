# Audit: 2025-01-liquid-ron

## Broken `onlyOperator` access control locks out operators and can freeze withdrawals
- Location: src/LiquidRon.sol : `onlyOperator` modifier
- Mechanism: The modifier reads `if (msg.sender != owner() || operator[msg.sender]) revert ErrInvalidOperator();`. The intended check is "revert unless caller is the owner **or** a registered operator" (`if (msg.sender != owner() && !operator[msg.sender])`). As written, the only way to pass is `msg.sender == owner() && operator[msg.sender] == false`: every address registered via `updateOperator(...)` always reverts, and if the owner is ever also flagged as an operator the owner is locked out too. This gates `harvest`, `harvestAndDelegateRewards`, `delegateAmount`, `redelegateAmount`, `undelegateAmount`, and `finaliseRonRewardsForEpoch`.
- Impact: Designated operators can never run the protocol, and because `finaliseRonRewardsForEpoch` is the only way to settle the `requestWithdrawal` queue, adding the owner as an operator (or relying on operators) permanently bricks reward harvesting, (re/un)delegation, and queued user withdrawals, stranding staked funds.

## `totalAssets()` double-counts accrued operator fees, inflating share price
- Location: src/LiquidRon.sol : `totalAssets` / `harvest` / `fetchOperatorFee`
- Mechanism: `totalAssets()` returns `super.totalAssets() + getTotalStaked() + getTotalRewards()`, where `super.totalAssets()` is the vault's raw WRON balance and `getTotalRewards()` already subtracts the operator fee from *unclaimed* rewards. When `harvest` runs, the proxy deposits the **gross** claimed amount into the vault as WRON (raising the WRON balance), the corresponding validators’ rewards drop to zero in `getTotalRewards()`, and `operatorFeeAmount += harvestedAmount * operatorFee / BIPS`. The fee portion now sits physically in the vault’s WRON balance and is counted in NAV, yet it is simultaneously a liability owed to `feeRecipient` (`operatorFeeAmount`) that is never subtracted from `totalAssets()`. Share price therefore jumps up by the fee on every harvest and drops back only when `fetchOperatorFee` removes that WRON.
- Impact: Depositors’ shares are over-valued by `operatorFeeAmount`, so redeemers (especially the last ones out) can withdraw the WRON earmarked as operator fees, leaving `fetchOperatorFee` underfunded and effectively transferring the protocol fee from the fee recipient to whoever exits first.

## Operator fee accrues on delegated rewards that never become liquid, creating an unbacked liability
- Location: src/LiquidRon.sol : `harvestAndDelegateRewards` (with `fetchOperatorFee`)
- Mechanism: `harvestAndDelegateRewards` increments `operatorFeeAmount += harvestedAmount * operatorFee / BIPS` using the amount returned by `LiquidProxy.harvestAndDelegateRewards`, but that path calls `delegateRewards` on the Ronin staking contract, which **re-stakes** the rewards — no RON is ever returned to the proxy or wrapped into the vault. The freshly delegated rewards are reflected as staked NAV via `getTotalStaked()`, while the accrued fee has no corresponding liquid WRON anywhere in the vault. When `feeRecipient` later calls `fetchOperatorFee`, the `_withdrawRONTo` must source the WRON from the vault’s general liquid reserve (user deposits / other harvested rewards).
- Impact: Operator fees on delegated rewards are paid out of depositors’ liquid principal rather than from the rewards themselves, draining the reserve needed to service `withdraw`/`redeem` and pushing the vault toward insolvency for liquid withdrawals.
