# Audit: 2024-04-noya

## Morpho position TVL adds borrowed debt instead of subtracting it
- Location: contracts/connectors/MorphoBlueConnector.sol : `_getPositionTVL`
- Mechanism: For a `MORPHO_POSITION_ID` position the connector computes `tvl = _getValue(loanToken, base, supplyAmount + borrowAmount + convertCToL(collateral, ...))`. `borrowAmount` (`pos.borrowShares.toAssetsUp(...)`) is the account's outstanding debt and must be *subtracted* from position value (net = supply + collateral − borrow); here it is added. Any vault position created via the connector's `borrow()` therefore reports a value inflated by `2 × borrowAmount`. This value flows into `TVLHelper.getTVL` → `AccountingManager.totalAssets()`/`TVL()`, which back the ERC4626 share price used by `previewRedeem`/`previewDeposit` and by `getProfit()`.
- Impact: A shareholder can redeem against the inflated TVL (via the withdraw queue priced by `previewRedeem`) to extract more base token than their shares are worth, draining remaining holders, and the phantom profit also mints unearned performance-fee shares.

## Gauge-staked Aerodrome LP is excluded from position TVL
- Location: contracts/connectors/AerodromeConnector.sol : `stake` / `_getPositionTVL`
- Mechanism: `_getPositionTVL` values the position purely from `IERC20(pool).balanceOf(address(this))` (the connector's *unstaked* LP balance). `stake()` deposits LP into the Aerodrome gauge (`IGauge(gauge).deposit(liquidity, address(this))`) without registering any accounting, so once LP is staked the gauge holds it and `balanceOf(address(this))` drops to (near) zero, making the staked value invisible to TVL. (Other connectors such as Stargate/Curve explicitly add the staked/gauge balance; this one does not.) Worse, `withdraw()` removes the holding position whenever `IERC20(pool).balanceOf(address(this)) == 0`, which is true while everything is staked.
- Impact: While LP is staked the vault under-reports TVL, so an attacker can `deposit` during that window to mint shares far below fair value and redeem them for a windfall after the position is unstaked, extracting value from existing holders.

## `onlyVaultMaintainer` requires maintainer AND emergency, permanently bricking connector/position management
- Location: contracts/accountingManager/Registry.sol : `onlyVaultMaintainer` modifier
- Mechanism: The modifier reverts on `msg.sender != vaults[_vaultId].maintainer || hasRole(EMERGENCY_ROLE, msg.sender) == false`, i.e. it only passes when the caller is the maintainer *and also* holds `EMERGENCY_ROLE`. The sibling modifiers (`onlyVaultMaintainerWithoutTimeLock`, `onlyVaultGoverner`) correctly use `&&`, giving the intended "specific-role OR emergency" semantics; this one inverts the second operand. In a normal deployment the per-vault maintainer and the global emergency role are distinct addresses, so neither the maintainer nor the emergency account satisfies both conditions, leaving `addConnector`, `updateConnectorTrustedTokens`, and `removeTrustedPosition` uncallable forever.
- Impact: Permanent denial of service on connector/trusted-token/position administration, preventing strategy deployment and blocking the maintainer from de-risking or retiring positions.

## `calculateWithdrawShares` is callable mid-execution, corrupting withdraw-group accounting and locking withdrawals
- Location: contracts/accountingManager/AccountingManager.sol : `calculateWithdrawShares` (and `executeWithdraw`'s `delete currentWithdrawGroup`)
- Mechanism: The guard only blocks when `isFullfilled == false && isStarted == true`. After `fulfillCurrentWithdrawGroup` sets `isFullfilled = true` (group still being executed, `isStarted` still true, `totalCBAmount` reset to 0), the guard passes, so a new batch can be calculated: it advances `withdrawQueue.middle` past those requests and adds their assets to `currentWithdrawGroup.totalCBAmount`. When `executeWithdraw` finishes the original group (`lastId == first`) it runs `delete currentWithdrawGroup`, wiping that freshly-accumulated `totalCBAmount`. The newly calculated requests now sit below `middle` with no group accounting; the next `startCurrentWithdrawGroup`/`fulfillCurrentWithdrawGroup` proceed with `totalCBAmount == 0`, yielding `totalABAmount = 0` and `totalCBAmountFullfilled = 0`, so `executeWithdraw` computes `data.amount * 0 / 0` and reverts on every call.
- Impact: Those users' withdrawals can never execute and their shares stay pinned by `withdrawRequestsByAddress` (which also blocks transfers in `_update`), permanently locking the shares and the assets owed to them.

## `executeWithdraw` overstates `totalWithdrawnAmount` on partial fulfillment, inflating profit and fees
- Location: contracts/accountingManager/AccountingManager.sol : `executeWithdraw`
- Mechanism: When a group is only partially funded, `fulfillCurrentWithdrawGroup` sets `totalABAmount = availableAssets < totalCBAmountFullfilled`, and each user is paid the pro-rated `baseTokenAmount = data.amount * totalABAmount / totalCBAmountFullfilled` (less than `data.amount`). However the loop accumulates `processedBaseTokenAmount += data.amount` (the full pre-haircut figure) and then does `totalWithdrawnAmount += processedBaseTokenAmount`, so `totalWithdrawnAmount` records more than actually left the vault. `getProfit()` returns `TVL + totalWithdrawnAmount - totalDepositedAmount`, so the overstatement is booked as profit.
- Impact: Phantom profit lets `recordProfitForFee`/`collectPerformanceFees` mint performance-fee shares against value that was never distributed, diluting the remaining shareholders.
