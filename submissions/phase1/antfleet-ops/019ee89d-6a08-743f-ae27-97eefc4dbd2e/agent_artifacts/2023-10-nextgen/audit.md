# Audit: 2023-10-nextgen

## Reentrancy in `mint` bypasses per-wallet allowlist and public mint caps
- Location: MinterContract.sol : `NextGenMinterContract.mint` (effect performed in NextGenCore.sol : `NextGenCore.mint` / `_mintProcessing`)
- Mechanism: `mint()` enforces per-wallet quotas with `require(_maxAllowance >= gencore.retrieveTokensMintedALPerAddress(col, …) + _numberOfTokens)` (allowlist) and `require(gencore.retrieveTokensMintedPublicPerAddress(col, msg.sender) + _numberOfTokens <= viewMaxAllowance(col))` (public). The counters backing those checks (`tokensMintedAllowlistAddress` / `tokensMintedPerAddress`) are incremented inside `NextGenCore.mint` *after* `_mintProcessing` → `_safeMint(_mintTo)`, which calls `onERC721Received` on the recipient. A recipient contract (`_mintTo == msg.sender`) re-enters `NextGenMinterContract.mint` from that callback; because the counter for the in-flight mint has not yet been written, the quota check reads a stale, lower count and passes again. Recursing through the callback mints many tokens despite a cap of 1. There is no `ReentrancyGuard`, and checks-effects-interactions is violated. The same stale read defeats the `salesOption == 3` "1 mint/period" pacing, since `lastMintDate[col]` is only written after the mint loop.
- Impact: A single address can mint far beyond its `_maxAllowance` / `maxCollectionPurchases` (up to remaining supply), draining a collection's allocation and destroying the per-wallet distribution limits.

## Auction bids can be cashed out twice at the end-time block (`claimAuction` never deactivates paid bids)
- Location: AuctionDemo.sol : `auctionDemo.claimAuction` (with `cancelBid` / `cancelAllBids`)
- Mechanism: In `claimAuction`, both the winner branch and the losing-bidder refund branch send ETH via raw `call` but never set `auctionInfoData[_tokenid][i].status = false`, so a bid remains "active" after it has been paid. `claimAuction` requires `block.timestamp >= getAuctionEndTime` while `cancelBid`/`cancelAllBids` require `block.timestamp <= getAuctionEndTime`, so in the block where `timestamp == auctionEndTime` all are callable, and the contract has no reentrancy guard. A bidder can therefore be refunded by `claimAuction` and then call `cancelBid` on the same still-`status==true` bid for a second refund (achievable either by re-entering during the refund `call` / the winner's `onERC721Received`, or simply with a second transaction in the same end-time block). The winner can re-enter to `cancelBid` their winning bid while still receiving the NFT, so the contract pays `highestBid` to both the winner and `owner()`.
- Impact: At the auction-end block an attacker collects their bid (or the winning bid) twice, stealing ETH that belongs to other bidders/the proceeds while keeping the NFT.

## `claimAuction` couples the NFT transfer with all refunds, permanently locking bidder funds
- Location: AuctionDemo.sol : `auctionDemo.claimAuction`
- Mechanism: `claimAuction` calls `IERC721(gencore).safeTransferFrom(ownerOfToken, highestBidder, _tokenid)` inside the same loop that issues every bidder refund, and `ownerOfToken` is read live from `ownerOf(_tokenid)`. If that transfer reverts — the current owner never approved the auction contract, transferred the token away, or the winning bidder is a contract whose `onERC721Received` reverts — the entire `claimAuction` reverts. After `auctionEndTime`, `cancelBid`/`cancelAllBids` are disabled (they require `timestamp <= getAuctionEndTime`), and there is no separate withdraw path, so losing bidders have no way to recover their ETH.
- Impact: A missing approval, a moved token, or a winning contract that rejects ERC-721 transfers bricks settlement and permanently locks every bidder's ETH in the contract.

## Predictable on-chain randomness allows token/trait grinding
- Location: RandomizerNXT.sol : `NextGenRandomizerNXT.calculateTokenHash`; XRandoms.sol : `randomPool.randomNumber` / `randomWord`
- Mechanism: The token hash that seeds each NFT's generative output is `keccak256(abi.encodePacked(_mintIndex, blockhash(block.number-1), randoms.randomNumber(), randoms.randomWord()))`, and `randomNumber()`/`randomWord()` are derived solely from `block.prevrandao`, `blockhash(block.number-1)`, and `block.timestamp`. Every input is known or influenceable at mint time, so a caller can precompute the hash that the next `mintIndex` will receive and decide whether (or in which block) to mint, and a block proposer can reorder/withhold to force a chosen outcome.
- Impact: Minters can selectively mint only tokens that resolve to rare/high-value traits, defeating the fairness and economic integrity of the randomized assignment.
