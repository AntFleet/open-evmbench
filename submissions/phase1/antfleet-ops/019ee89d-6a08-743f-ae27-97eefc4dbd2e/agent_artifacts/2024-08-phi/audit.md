# Audit: 2024-08-phi

## PhiFactory pays mint fees from its own balance, allowing free mints and ETH drain
- Location: src/PhiFactory.sol : `claim` (and `batchClaim`)
- Mechanism: `claim` computes `mintFee = getArtMintFee(artId, quantity_)` and then invokes `this.merkleClaim{ value: mintFee }(...)` / `this.signatureClaim{ value: mintFee }(...)`. The `{ value: mintFee }` is drawn from the **factory's own balance**, and `claim` never checks that the incoming `msg.value` covers `mintFee`. Because the inner self-call now sees `msg.value == mintFee`, it passes `_validateAndUpdateClaimState`'s `if (msg.value < getArtMintFee(...)) revert InvalidMintFee()` check, and `_processClaim` forwards the fee to `art.artAddress` and `protocolFeeDestination` out of factory funds; the caller's `msg.value` is ignored except that it adds to the balance. `batchClaim` has the identical structure (it forwards `ethValue_[i]` to `claim`, which still spends `mintFee` from balance). The factory routinely holds ETH because `createArt` forwards `msg.value` to the NFT, whose `createArtFromFactory` refunds `msg.value - artFee` back to the factory (`_msgSender()` there is the factory).
- Impact: Anyone holding any valid signer signature or Merkle proof can call `claim`/`batchClaim` with `msg.value = 0` and repeatedly mint for free, draining all ETH held by the PhiFactory.

## Claim flow never blocks repeat claims and does not bind `quantity` to the authorization
- Location: src/PhiFactory.sol : `_validateAndUpdateClaimState` (with `signatureClaim` / `merkleClaim`)
- Mechanism: `_validateAndUpdateClaimState` sets `credMinted[...][minter_] = true` and `artMinted[artId_][minter_] = true` but never reverts when they are already `true` (the `AddressAlreadyMinted` error is defined but unused). The signed payload for `signatureClaim` (`abi.encode(expiresIn_, minter_, ref_, verifier_, artId_, chainid, data_)`) and the Merkle leaf (`keccak256(abi.encode(minter_, leafPart_))`) do **not** include `quantity_`; quantity is taken from the unsigned `mintArgs_`. There is also no nonce on the signature and no expiry on a Merkle proof. The only cap is `art.numberMinted + quantity_ > art.maxSupply`.
- Impact: A single eligible address can replay one signature/Merkle proof with an arbitrary `quantity_` to mint an art's entire `maxSupply`, denying all other eligible users and breaking the one-claim-per-credential invariant (and, combined with the `claim` balance-funding bug, doing so for free).

## Cred bookkeeping mutators are `public` (missing access control)
- Location: src/Cred.sol : `_addCredIdPerAddress` and `_removeCredIdPerAddress`
- Mechanism: These intended-internal helpers are declared `public` and mutate `_credIdsPerAddress[sender_]`, `_credIdsPerAddressCredIdIndex`, and `_credIdsPerAddressArrLength` for an arbitrary `sender_` with no caller restriction. An attacker can push arbitrary credIds onto a victim's array (`_addCredIdPerAddress`) or remove entries (`_removeCredIdPerAddress`), desynchronising the index bookkeeping from `shareBalance`. Crucially, the legitimate sell-to-zero path in `_updateCuratorShareBalance` calls `_removeCredIdPerAddress`; once an attacker has corrupted the stored index for a victim+credId, that internal call hits `if (credId_ != credIdToRemove) revert WrongCredId()` or `revert IndexOutofBounds()`.
- Impact: Anyone can corrupt any curator's position accounting and permanently brick a victim's ability to sell their last share of a cred (griefing/DoS), and arbitrarily inflate the per-address arrays.

## Missing reentrancy guard on single trades bypasses the share-lock period
- Location: src/Cred.sol : `_handleTrade` (buy branch / `lastTradeTimestamp`)
- Mechanism: `_handleTrade` carries only `whenNotPaused` — the `nonReentrant` guard exists but is applied solely to `_executeBatchTrade`. On a buy, `shareBalance` and `cred.currentSupply` are updated, then `_msgSender().safeTransferETH(excessPayment)` is called, and only *after* that external call is `lastTradeTimestamp[credId_][curator_] = block.timestamp` written. A buyer deliberately overpays so `excessPayment > 0`, then re-enters during the refund: `lastTradeTimestamp` still holds its old (or zero) value, so the sell-path check `block.timestamp <= lastTradeTimestamp[credId_][curator_] + SHARE_LOCK_PERIOD` passes and the just-bought shares can be sold in the same transaction. The re-entrant callback can also invoke `CuratorRewardsDistributor.distribute(credId)` first, crediting the attacker's freshly bought shares before selling.
- Impact: An attacker can atomically buy → trigger curator-reward distribution → sell, skimming a proportional share of the curator reward pool while only momentarily holding shares, defeating the 10-minute `SHARE_LOCK_PERIOD` anti-flip protection.

## `buyShareCredFor` lets a third party reset a victim's lock timer
- Location: src/Cred.sol : `buyShareCredFor` → `_handleTrade` (buy branch)
- Mechanism: `buyShareCredFor` routes to `_handleTrade(..., curator_, ...)`, whose buy branch unconditionally sets `lastTradeTimestamp[credId_][curator_] = block.timestamp` and increases `curator_`'s share balance, with no consent from `curator_`. An attacker repeatedly buys 1 share "for" a target curator (paying the curve price) shortly before the curator wants to exit, continually resetting their `lastTradeTimestamp`, so the curator's `sellShareCred` always reverts with `ShareLockPeriodNotPassed`.
- Impact: A griefer can indefinitely prevent a targeted curator from selling their shares (and force unwanted shares onto them).

## Zero-balance curators are never pruned, enabling permanent distribution DoS
- Location: src/Cred.sol : `_updateCuratorShareBalance` (sell branch) → src/reward/CuratorRewardsDistributor.sol : `distribute`
- Mechanism: On a full sell, `_updateCuratorShareBalance` calls `shareBalance[credId_].set(sender_, currentNum - amount_)` with value `0` instead of `EnumerableMap.remove`, so every address that has ever held the cred remains a permanent map entry. `distribute` calls `credContract.getCuratorAddresses(credId, 0, 0)` → `_getCuratorData`, which iterates over `shareBalance[credId_].length()` (all historical holders, including zero-balance), then `distribute` loops again and `depositBatch` loops once more. The holder set only grows; an attacker cycling many funded addresses (buy → wait lock → sell to zero), or organic churn over time, pushes the iteration past the block gas limit.
- Impact: `distribute(credId)` eventually reverts permanently, locking all curator-reward ETH (`balanceOf[credId]`) deposited for that cred with no recovery path.
